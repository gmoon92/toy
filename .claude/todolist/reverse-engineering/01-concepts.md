# Reverse Engineering 개념 정의

## 1. 핵심 용어

### Reverse Engineering (역설계/역공학)

기존 시스템의 동작, 구조, 설계 의도를 분석하여 이해하는 과정.
코드에서 설계/문서로 거꾸로 올라가는 흐름을 의미함.

**적용 상황**
- 기존 프로젝트의 기능, 구조, 시스템 설계를 문서화할 때
- 코드 기반으로 설계 의도를 파악할 때

---

## 2. 유사 개념 비교

| 용어 | 초점 | 적용 상황 |
|------|------|----------|
| **System Analysis** | 기능, 흐름, 데이터 구조 | 기능 중심 분석 |
| **Architecture Recovery** | 레이어, 모듈, 의존성 재구성 | 설계 문서가 없는 레거시 분석 |
| **Program Comprehension** | 코드 이해 전체 과정 | 학술/이론적 맥락 |
| **Software Archaeology** | 오래된 프로젝트 히스토리 추적 | 레거시 분석 (비유적 표현) |
| **Codebase Audit** | 코드 품질, 구조 검토 | 리팩토링/개선 전 검토 |

---

## 3. Reverse Engineering vs Architecture Recovery

둘은 겹치는 영역이 있으나 관점과 산출물 수준이 다름.

### Reverse Engineering

| 항목 | 내용 |
|------|------|
| **목적** | 코드 기반으로 동작, 로직, 의도 파악 |
| **질문 관점** | 이 함수는 왜 존재하는가? 데이터는 어디서 와서 어디로 가는가? |
| **대상 범위** | 함수, 클래스, API, DB 쿼리 등 미시적 단위 |
| **결과물** | 코드 이해, 로직 흐름 정리, 기능 단위 문서 |

**한 줄 요약**: 코드를 읽어서 이해하는 단계

### Architecture Recovery

| 항목 | 내용 |
|------|------|
| **목적** | 흩어진 코드를 기반으로 전체 구조 재구성 |
| **질문 관점** | 시스템은 몇 개의 레이어로 나뉘는가? 모듈 간 의존성은? |
| **대상 범위** | 모듈, 레이어, 서비스, 컴포넌트 간 관계 (거시적) |
| **결과물** | 아키텍처 다이어그램, 모듈 구조도, 시스템 설계 문서 |

**한 줄 요약**: 이해한 내용을 설계 수준으로 끌어올리는 단계

### 두 개념의 관계

```
Reverse Engineering -> Architecture Recovery
      (이해)               (구조화)
```

- RE 없이 AR은 불가능 (이해가 선행되어야 함)
- AR 없이 RE는 "그냥 코드 읽은 상태"에 머무름

**실무적 차이**

| 단계 | 상태 |
|------|------|
| RE만 한 경우 | "이 코드 왜 이렇게 짰는지 알겠다" - 그러나 전체 구조 설명 불가, 리팩토링 방향 미정 |
| AR까지 한 경우 | "이 시스템은 이런 레이어 구조고, 여기서 문제가 생긴다" - 리팩토링/신규 기능 설계 가능 |

---

## 4. 실무 표현 추천

문서나 PR, 발표에서 사용하기 적합한 표현:

- "기존 시스템에 대한 Reverse Engineering 및 Architecture Recovery 수행"
- "레거시 코드 기반 아키텍처 복원 및 문서화"
- "기존 서비스 시스템 분석 및 구조 재정의"

---

## 5. 요약

| 구분 | 정의 |
|------|------|
| Reverse Engineering | 이해 (코드의 동작과 의미 파악) |
| Architecture Recovery | 구조화 + 설계화 (이해를 바탕으로 설계 수준으로 끌어올림) |

두 개념은 경쟁 관계가 아닌 연속된 단계이며, 기능/구조/설계 문서화를 위해서는 RE를 기반으로 AR까지 수행하는 것이 필요함.
