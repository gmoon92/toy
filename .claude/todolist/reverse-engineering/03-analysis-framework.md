# Reverse Engineering 분석 프레임워크

기존 프로젝트를 체계적으로 분석하여 문서화하는 프레임워크.

---

## 1. 목표 설정

### 핵심 목표

기존 프로젝트를 체계적으로 분석하여 기능, 구조, 시스템 설계를 문서화.

### 구체적인 니즈

1. **올바른 용어 확보**: "역설계"의 적절성, 더 정확한 용어 확인
2. **개념적 구분**: Reverse Engineering vs Architecture Recovery 차이 이해
3. **실행 방법론**: 실제 프로젝트에 적용할 수 있는 구체적인 단계와 방법

---

## 2. 대상 상황 분석

### 적용 가능한 시나리오

| 시나리오 | 설명 |
|----------|------|
| 레거시 프로젝트 인수인계 | 문서가 부족한 기존 시스템 파악 |
| 리팩토링 준비 | 구조 개선 전 현재 상태 문서화 |
| 신규 팀원 온볼링 | 팀원들이 시스템을 이해할 수 있는 문서 작성 |
| 아키텍처 검토 | 현재 시스템의 구조적 문제점 파악 |

### 공통 필요성

- "코드 읽기"를 넘어서는 "체계적 분석"
- 추상화 수준별로 다른 관점의 문서
- 팀과 공유 가능한 형태의 산출물

---

## 3. 실행 계획 템플릿

### Phase 1: 범위 정의 (1-2시간)

1. **대상 프로젝트 선정**: 어떤 프로젝트/모듈을 분석할 것인가
2. **분석 깊이 결정**:
   - Level 1: High-level 아키텍처만
   - Level 2: 주요 모듈과 의존성
   - Level 3: 클래스/함수 수준의 상세 분석
3. **산출물 형식 결정**: 다이어그램, 문서, 둘 다

[02-methodology.md](02-methodology.md) 참조.

## 4. 문서화 전략

### 문서의 역할

문서는 "설명"이 아닌 "의사결정 도구"여야 함. 개선 포인트 포함이 필수.

### 권장 섹션 구성

| 섹션 | 내용 | 중요도 |
|------|------|--------|
| Overview | 시스템 목적, 주요 기능 요약 | 필수 |
| Architecture Overview | 전체 구조 다이어그램, 레이어 설명 | 필수 |
| Module Structure | 모듈 목록, 책임, 의존성 | 필수 |
| Component Detail | 각 컴포넌트별 역할, 메서드, 의존성, 입출력 | 권장 |
| Data Model | ERD, 주요 테이블 설명, 엔티티 관계 | 권장 |
| Flow | 주요 시나리오별 처리 흐름, Sequence Diagram | 권장 |
| Integration | 외부 시스템 연동, API/MQ/Cache | 필요시 |
| Technical Decisions | 왜 이렇게 설계됐는지 추론, 문제점, 개선 포인트 | 핵심 |

## 5. 사전 확인 사항

구체적인 계획 수립을 위해 확인이 필요한 항목:

1. **대상 프로젝트는 무엇인가?** (현재 프로젝트인가, 다른 프로젝트인가)
2. **분석의 목적은 무엇인가?** (리팩토링, 인수인계, 단순 학습 등)
3. **얼마나 깊이 분석해야 하는가?** (아키텍처 수준 vs 코드 수준)
4. **누가 이 문서를 읽을 것인가?** (본인, 팀원, 경영진 등)
5. **제한 시간이 있는가?** (급한가, 여유로운가)

## 7. 요약

Reverse Engineering 및 Architecture Recovery를 통한 체계적인 프로젝트 분석은 다음 순서로 진행:

1. **목표와 범위 정의**: 왜 분석하는가, 어디까지 분석하는가
2. **코드 분석 (RE)**: 동작과 의미 파악
3. **흐름 모델링**: 시각적 재구성
4. **구조화 (AR)**: 설계 수준으로 끌어올리기
5. **문서화**: 의사결정 도구로서의 문서 작성
6. **검증**: 문서와 실제 동작의 일치 확인
