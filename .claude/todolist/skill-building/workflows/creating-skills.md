# 새 스킬 생성 (Creating New Skills)

처음부터 새 Claude Code 스킬을 생성하기 위한 가이드입니다.

---

## 시작하기 전에

### 전제조건

1. **필요성 식별**: 반복적인 컨텍스트나 워크플로우가 스킬로 혜택을 받을 명확한 사용 사례가 있음
2. **스킬 없이 테스트**: 먼저 일반적인 프롬프팅으로 Claude와 함께 작업을 해결해 보세요
3. **패턴 식별**: 반복적으로 제공하는 정보를 기록하세요
4. **기존 스킬 확인**: 유사한 기능이 이미 존재하지 않는지 확인하세요

### 핵심 질문

- 이 스킬이 해결하는 특정 문제는 무엇인가?
- Claude에게 필요한 컨텍스트나 지식은 무엇이며, 그것이 없는 것은 무엇인가?
- 이 스킬이 유지보수를 정당화할 만큼 자주 사용될 것인가?
- 스킬의 범위를 명확히 정의할 수 있는가?

---

## 단계 1: 스킬 계획

### 범위 정의

**스킬이 할 것:**
- 3-5개 핵심 기능 나열
- 주요 사용 사례 식별
- 성공 기준 정의

**스킬이 하지 않을 것:**
- 범위 밖 작업을 명시적으로 나열
- 범위 확장 방지를 위한 경계 정의

### 실행 스타일 선택

**높은 자율성** (에이전트가 대부분의 작업 수행):
- 여러 단계가 있는 복잡한 워크플로우
- 분석과 의사결정이 필요함
- 예시: 커밋 스킬, 코드 리뷰

**낮은 자율성** (참조 중심):
- 지식 또는 API 문서
- 스타일 가이드와 규칙
- 예시: 라이브러리 문서, 코딩 표준

**도구 기반** (특정 작업 실행):
- 결정적 작업
- 파일 처리, 데이터 변환
- 예시: PDF 처리, 스프레드시트 조작

## 단계 2: 디렉토리 구조 생성

### 기본 구조

```
skill-name/
├── SKILL.md              # 주요 스킬 문서
├── references/           # 상세 참조 (선택사항)
│   ├── process/         # 단계별 프로세스
│   ├── guides/          # 하우투 가이드
│   └── support/         # 지원 문서
├── scripts/             # 실행 가능한 스크립트 (선택사항)
│   ├── README.md
│   └── [categories]/
└── templates/           # UI 템플릿 (로드되지 않음, 선택사항)
```

### 디렉토리 생성

```bash
# skills 디렉토리에서
mkdir -p skill-name/references/{process,guides,support}
mkdir -p skill-name/scripts
mkdir -p skill-name/templates
```

## 단계 3: SKILL.md 작성

### 템플릿 사용

[skill-template.md](../templates/skill-template.md)로 시작하고 커스터마이즈:

### 필수 섹션

**1. 프론트매터**
```yaml
---
name: skill-name
description: 간략한 설명 (200자 미만, 구체적으로)
disable-model-invocation: false
user-invocable: true
allowed-tools: Read, Grep, Glob, Bash, AskUserQuestion, Write, Edit
---
```

**핵심 필드:**

**`name` (필수):**
- 동명사 형태(동사-ing) 사용: `processing-pdfs`, `analyzing-data`
- **검증 규칙:**
  - 최대 64자
  - 소문자, 숫자, 하이픈만 포함
  - XML 태그 포함 불가
  - 예약어 불가: "anthropic", "claude"

**`description` (필수):**
- Claude가 인식해야 할 주요 용어 포함, 언제 사용하는지 설명
- **검증 규칙:**
  - 비어 있으면 안 됨
  - 최대 1024자
  - XML 태그 포함 불가

**`user-invocable`:**
- 사용자가 `/skill-name`으로 호출해야 하면 `true`로 설정

**`allowed-tools`:**
- 스킬에 필요한 도구만 제한

**2. 개요**
- 간략한 목적 설명 (1-2문장)
- 핵심 기능 (3-5개 글머리)
- 범위 정의 (하는 것과 하지 않는 것)

**3. 참조 로딩 시기**
```markdown
### 단계 1: [단계 이름]
- **로드**: [reference/path.md](references/reference/path.md)
- **목적**: 이 단계가 하는 것
- **언제**: 이 단계를 실행할 때

### 단계 2: [단계 이름] (조걶)
- **로드**: [reference/path.md](references/reference/path.md)
- **목적**: 이 단계가 하는 것
- **언제**: [조건]일 때만
```

**이것이 중요한 이유:**
- 모든 참조를 미리 로드하는 것 방지
- 50-70% 컨텍스트 토큰 절약
- 스킬을 더 빠르고 효율적으로 만듦

**4. 핵심 원칙**
- 2-4개 기본 원칙 나열
- 각각이 왜 중요한지 설명
- 간결하게 유지 (각 2-3줄)

**5. 빠른 참조**
- Claude가 즉시 필요로 하는 필수 정보
- 형식, 패턴, 핵심 개념
- 생각해 보세요: "Claude가 작업을 시작하는 데 무엇이 필요한가?"

**6. 실행 흐름**
```markdown
## 실행 흐름

```
1. 단계 1     → [참조](references/step1.md)
2. 단계 2     → [참조](references/step2.md) (필요시)
3. 단계 3     → [참조](references/step3.md)
```
```

**7. 사용 예시**
```markdown
## 사용법

**스킬 호출:**
```
/skill-name
```

**자연어:**
```
예시 명령어 1
예시 명령어 2
예시 명령어 3
```
```

### 간결하게 유지

**목표: SKILL.md 200-500줄**

SKILL.md가 500줄을 초과하면:
- 세부사항을 참조 문서로 이동
- 주요 섹션에서 참조로 링크
- SKILL.md에는 필수 정볧만 유지

## 단계 4: 참조 문서 생성

### 참조를 생성할 때

**프로세스 문서** (`references/process/`):
- 단계별 워크플로우
- 다단계 작업
- 순차적 절차

**가이드** (`references/guides/`):
- 하우투 지시문
- 모범 사례
- 의사결정 프레임워크

**지원 문서** (`references/support/`):
- 예시와 템플릿
- 문제 해결 가이드
- FAQ

### 참조 문서 구조

```markdown
# 문서 제목

간략한 개요 (1-2문장).

---

## 목적

이 문서가 다루는 것과 언제 사용하는지 설명.

## 내용

### 섹션 1
상세 내용...

### 섹션 2
상세 내용...

## 예시

개념을 보여주는 구체적인 예시.

## 관련 문서

- [관련 문서 1](path/to/doc1.md)
- [관련 문서 2](path/to/doc2.md)
```

### 작성 지침

**간결하게:**
- Claude가 똑똄하다고 가정
- 명백한 개념을 과도하게 설명하지 마세요
- Claude가 실제로 필요로 하는 것에 집중

**예시 사용:**
- 보여주기, 그냥 말하지 않기
- 입출력 쌍 포함
- 패턴을 명확하게 보여주기

**참조 링크:**
- 링크를 한 단계 깊이로 유지
- 참조 체인 생성하지 마세요
- 탐색을 명확하게 만드세요

## 단계 5: 스크립트 추가 (필요시)

### 스크립트를 추가할 때

**스크립트를 사용할 때:**
- 알고리즘이 복잡하거나 오류가 발생하기 쉬움
- 실행이 결정적이어야 함
- 작업이 자주 반복됨
- 토큰 효율성이 중요함

**스크립트를 사용하지 말아야 할 때:**
- Claude가 처리할 수 있는 간단한 작업
- 유연성이 일관성보다 중요함
- 스크립트가 작업보다 복잡함

### 스크립트 지침

**1. 책임별로 구성:**
```
scripts/
├── README.md           # 모든 스크립트 문서화
├── analysis/          # 데이터 수집 및 분석
├── algorithms/        # 감지 및 의사결정 로직
├── generation/        # 콘텐츠 생성
├── validation/        # 입출력 검증
├── execution/         # 주요 작업
└── utils/             # 도우미 함수
```

**2. 표준 I/O 사용:**
- stdin을 통한 입력 (JSON 형식)
- stdout을 통한 출력 (JSON 형식)
- stderr를 통한 오류

**3. 강력한 지시문 추가:**
```markdown
**필수: 미리 빌드된 스크립트 사용 (인라인 명령어 금지)**

```bash
# EXECUTE_SCRIPT: scripts/category/script_name.sh

./scripts/category/script_name.sh < input.json > output.json
```

**중요:**
- 문서에서 명령어 재구성하지 마세요
- 항상 `EXECUTE_SCRIPT:` 지시문을 통해 미리 빌드된 스크립트 사용
```

**4. README에 문서화:**
자세한 내용은 [script-extraction.md](script-extraction.md) 참조.

## 단계 6: 스킬 테스트

### 테스트 체크리스트

**기능:**
- [ ] 스킬이 예상대로 활성화됨
- [ ] 모든 참조가 올바르게 로드됨
- [ ] 스크립트가 오류 없이 실행됨
- [ ] 출력이 예상과 일치함

**문서:**
- [ ] 모든 링크가 올바르게 해결됨
- [ ] 예시가 보여진 대로 작동함
- [ ] 지시문이 명확함
- [ ] 깨진 참조 없음

**토큰 효율성:**
- [ ] SKILL.md 500줄 미만
- [ ] 참조가 점진적으로 로드됨
- [ ] 불필요한 장황함 없음
- [ ] 스크립트가 토큰 사용을 줄임

### 여러 모델로 테스트

- **Haiku**: 충분한 안내가 있는가?
- **Sonnet**: 명확하고 효율적인가?
- **Opus**: 과도하게 설명되지 않았는가?

### 실제 사용 사례 테스트

1. 3-5개 현실적인 시나리오 생성
2. 새로운 컨텍스트(새 대화)에서 스킬 테스트
3. 동작이 예상과 일치하는지 확인
4. 관찰에 기반하여 개선

## 단계 7: 개선 및 반복

### 일반적인 문제

**스킬이 활성화되지 않음:**
- 설명에 더 많은 트리거 키워드 추가
- 사용 사례를 더 명시적으로 만들기
- user-invocable 설정 확인

**너무 장황함:**
- 세부사항을 참조로 이동
- 불필요한 설명 제거
- Claude가 기본을 알고 있다고 가정

**Claude가 스크립트를 무시함:**
- 더 강력한 지시문 추가 (필수, 스크립트 실행)
- 명시적인 금지 사용 (인라인 금지)
- README에서 스크립트를 명확하게 문서화

**참조가 로드되지 않음:**
- 상대 경로 확인
- 파일명이 정확히 일치하는지 확인
- 링크를 수동으로 테스트

### 반복 워크플로우

1. **실제 시나리오에서** 스킬 사용
2. **동작 관찰**: 효과적인 것과 그렇지 않은 것 기록
3. **문제 식별**: Claude가 어려워하는 곳은 어디인가?
4. **대상 변경**: 특정 문제 수정
5. **다시 테스트**: 개선 확인
6. **반복**: 만족스러울 때까지 반복

## 예시 워크플로우

**시나리오:** 코드 리뷰를 위한 스킬 생성

**1. 계획:**
```
목적: 체계적인 코드 리뷰를 위해 Claude 안내
핵심 기능:
- 코드 구조와 조직 분석
- 잠재적 버그와 엣지 케이스 식별
- 가독성 개선 제안
- 보안 취약점 확인
범위: 품질에 집중, 스타일 선호도는 제외
```

**2. 구조:**
```bash
mkdir -p code-review/references/{process,guides,support}
mkdir -p code-review/scripts/validation
```

**3. SKILL.md:**
```yaml
---
name: code-review
description: 버그, 보안 문제, 품질 개선을 위해 코드를 체계적으로 검토합니다. PR 검토나 코드 품질 분석 시 사용.
user-invocable: true
allowed-tools: Read, Grep, Glob, AskUserQuestion
---

## 개요
...
```

**4. 참조:**
- `references/process/review-checklist.md` - 단계별 검토 프로세스
- `references/guides/security-patterns.md` - 일반적인 보안 문제
- `references/support/examples.md` - 예시 리뷰

**5. 테스트:**
- 3개의 다른 코드베이스에서 테스트
- 모든 검사가 올바르게 실행되는지 확인
- 관찰에 기반하여 개선

**6. 반복:**
- 누락된 보안 검사 추가
- 제안의 명확성 개선
- 토큰 사용 최적화
