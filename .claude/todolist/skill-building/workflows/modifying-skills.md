# 기존 스킬 수정 (Modifying Existing Skills)

기존 Claude Code 스킬을 리팩토링하고 개선하기 위한 가이드입니다.

---

## 스킬을 수정해야 할 때

### 일반적인 트리거

**성능 문제:**
- 스킬이 너무 많은 컨텍스트를 미리 로드
- 실행하는 데 너무 오래 걸림
- 불필요한 콘텐츠에 토큰 낭비

**유지보수 부담:**
- 알고리즘이나 로직 업데이트가 어려움
- 변경사항이 여러 파일 수정을 요구
- 문서 전체에 코드 중복

**동작 문제:**
- Claude가 지시문을 일관되게 따르지 않음
- 스킬이 활성화되어야 할 때 활성화되지 않음(또는 그 반대)
- 출력이 예측할 수 없거나 변동됨

**기능 추가:**
- 새로운 사용 사례가 등장
- 사용자가 추가 기능을 요청
- 관련 워크플로우에 통합 필요

---

## 수정 유형

### 1. 토큰 최적화

**문제:** 스킬이 모든 것을 미리 로드

**해결책:** 점진적 공개 구현

**단계:**
1. 항상 필요한 섹션과 조걶 섹션 분석
2. SKILL.md를 단계 기반 로딩으로 분할
3. 상세한 콘텐츠를 참조로 이동
4. "참조 로딩 시기" 섹션 추가

**예시:**

**이전:**
```markdown
## PDF 처리

[5000줄의 상세한 PDF 처리 지시문, 예시, 엣지 케이스, 문제 해결 등...]
```

**이후:**
```markdown
## PDF 처리

간략한 개요 (50줄)

### 참조 로딩 시기
- **단계 1**: [extraction.md](references/extraction.md) - 텍스트 추출만
- **단계 2**: [forms.md](references/forms.md) - 양식 채우기(조걶)
- **단계 3**: [advanced.md](references/advanced.md) - 복잡한 작업(조걶)
```

**영향:** 60-80% 토큰 감소

### 2. 스크립트 추출

**문제:** 문서의 인라인 코드

**해결책:** 실행 가능한 스크립트로 추출

자세한 가이드는 [script-extraction.md](script-extraction.md) 참조.

**빠른 체크리스트:**
- [ ] 모든 인라인 코드 블록 식별(bash, JavaScript, Python)
- [ ] 책임별로 구성하여 스크립트로 추출
- [ ] `EXECUTE_SCRIPT:` 지시문 추가
- [ ] scripts/README.md 문서 생성
- [ ] 문서에서 인라인 코드 제거

### 3. 결정론 개선

**문제:** Claude가 매번 지시문을 다르게 해석

**해결책:** 더 강력한 지시문과 명시적인 제약 사용

**약한 지시문:**
```markdown
출력을 확인하기 위해 검증 스크립트를 사용해야 합니다.
```

**강력한 지시문:**
```markdown
**필수:** 진행하기 전에 항상 검증 스크립트를 사용하세요.

```bash
# EXECUTE_SCRIPT: scripts/validation/validate.sh
./scripts/validation/validate.sh < output.json
```

**중요:**
- 검증이 실패하면 진행하지 마세요
- 검증 단계를 절대 건결하지 마세요
- 항상 스크립트 사용(인라인 명령어 금지)
```

**사용할 키워드:**
- `필수:` - 필수 동작
- `EXECUTE_SCRIPT:` - 스크립트 참조 마커
- `하지 마세요` - 명시적 금지
- `항상` - 강제
- `절대` - 강력한 금지
- `중요:` - 최고 우선순위
- `주목:` - 중요한 참고사항

### 4. 범위 세분화

**문제:** 스킬이 너무 많이(또는 너무 적게) 시도

**해결책:** 경계 명확화

**명시적인 범위 추가:**
```markdown
**범위:**
- ✅ 이 스킬이 하는 것
- ✅ 이 스킬이 처리하는 것
- ❌ 절대: 이 스킬이 피하는 것
- ❌ 절대: 범위 밖 작업
```

**설명 업데이트:**
```yaml
# 이전
description: 문서 도움

# 이후
description: Word 문서(.docx)를 생성하고 편집합니다. .docx 파일 작업 시 사용, PDF나 스프레드시트는 해당 안 됨.
```

### 5. 활성화 튜닝

**문제:** 스킬이 활성화되어야 할 때 활성화되지 않음(또는 잘못 활성화)

**해결책:** 설명에서 트리거 키워드 개선

**약한 설명:**
```yaml
description: 커밋 도움
```

**강력한 설명:**
```yaml
description: Git 변경사항을 분석하고 프로젝트 규칙에 따라 커밋 메시지를 생성합니다. 커밋을 생성하거나 사용자가 staging 변경사항, 커밋 메시지, 또는 git 커밋을 언급할 때 사용.
```

**팁:**
- 사용자가 말할 수 있는 모든 관련 키워드 포함
- 파일 유형이나 도구를 명시적으로 언급
- 무엇(what)과 언제(when) 사용하는지 모두 설명

**검증 규칙:**

`name` 필드(필수):
- 최대 64자
- 소문자, 숫자, 하이픈만 포함
- XML 태그 포함 불가
- 예약어 불가: "anthropic", "claude"
- 동명사 형태(동사-ing) 사용: `processing-pdfs`, `analyzing-data`

`description` 필드(필수):
- 비어 있으면 안 됨
- 최대 1024자
- XML 태그 포함 불가
- Claude가 인식해야 할 주요 용어 포함

## 리팩토링 워크플로우

### 단계 1: 현재 상태 평가

**스킬 분석:**
```bash
# 파일 크기 확인
find skill-name -name "*.md" -exec wc -l {} \; | sort -rn

# 인라인 코드 블록 찾기
grep -r "^\`\`\`bash" skill-name/
grep -r "^\`\`\`javascript" skill-name/
grep -r "^\`\`\`python" skill-name/

# 토큰 수 확인(대략적인 추정)
cat skill-name/git-commit.md | wc -w
```

**위험 신호:**
- SKILL.md 500줄 초과
- 여러 개의 큰 인라인 코드 블록
- 모든 콘텐츠가 미리 로드됨(단계 기반 로딩 없음)
- 약한 지시문 언어("you can", "consider", "might")

### 단계 2: 문제 식별

**문제 목록 생성:**

예시:
```
커밋 스킬의 문제:
1. SKILL.md에 15개 이상의 인라인 bash 스크립트(~2000 토큰 낭비)
2. 문서와 알고리즘 코드가 섞임
3. 단계 기반 로딩 없음(모든 것을 미리 로드)
4. Claude가 때로는 파일 대신 스크립트를 재구성
5. 알고리즘 변경을 위해 여러 문서 업데이트 필요
```

**우선순위:**
- 높은 영향, 낮은 노력 → 먼저 수행
- 높은 영향, 높은 노력 → 신중하게 계획
- 낮은 영향 → 미루거나 건결

### 단계 3: 변경 계획

**각 문제에 대해 결정:**
- 무엇을 변경해야 하는가
- 어떤 파일이 영향을 받는가
- 하위 호환성 우려사항
- 테스트 요구사항

**계획 문서화:**
```markdown
## 리팩토링 계획

### 1. 인라인 스크립트 추출
- 영향: step1-analysis.md, step5-execute.md
- 생성: scripts/{analysis,execution}/*.sh
- 업데이트: SKILL.md에 EXECUTE_SCRIPT 지시문
- 테스트: 스크립트 구문 검증, 실행 테스트

### 2. 점진적 로딩 구현
- 영향: SKILL.md
- 생성: "참조 로딩 시기" 섹션
- 업데이트: 단계 기반 문서 구조
- 테스트: 각 단계에서 로딩 검증

### 3. 지시문 강화
- 영향: 모든 참조 문서
- 추가: 필수, 하지 마세요, 항상 키워드
- 업데이트: 스크립트 실행 예시
- 테스트: Claude가 지시문을 따르는지 확인
```

### 단계 4: 변경 실행

**논리적 순서로 작업:**

1. **구조적 변경 먼저** (디렉토리, 파일 조직)
2. **콘텐츠 추출** (코드를 스크립트로 이동, 큰 파일 분할)
3. **문서 업데이트** (지시문, 참조, 예시)
4. **검증** (구문, 링크, 실행 테스트)

**버전 관리 사용:**
```bash
# 리팩토링을 위한 브랜치 생성
git checkout -b refactor/skill-name

# 점진적으로 커밋
git add -p
git commit -m "refactor(skill-name): 분석 스크립트 추출"
git commit -m "refactor(skill-name): 점진적 로딩 추가"
git commit -m "refactor(skill-name): 실행 지시문 강화"
```

### 단계 5: 변경 테스트

**기능 테스트:**
- [ ] 스킬이 올바르게 활성화됨
- [ ] 모든 단계가 예상대로 실행됨
- [ ] 스크립트가 오류 없이 실행됨
- [ ] 참조가 올바르게 로드됨
- [ ] 출력 품질이 유지되거나 개선됨

**토큰 테스트:**
- [ ] 이전/이후 토큰 사용량 측정
- [ ] 점진적 로딩이 작동하는지 확인
- [ ] 불필요한 콘텐츠가 로드되지 않는지 확인

**동작 테스트:**
- [ ] 새로운 대화 컨텍스트로 테스트
- [ ] Claude가 지시문을 따르는지 확인
- [ ] 결정론 확인(같은 입력 → 같은 출력)
- [ ] 엣지 케이스와 오류 시나리오 테스트

**크로스 모델 테스트:**
- [ ] Haiku로 테스트(충분한 안내?)
- [ ] Sonnet으로 테스트(명확하고 효율적?)
- [ ] Opus로 테스트(과도하게 설명되지 않음?)

### 단계 6: 변경 문서화

**변경 로그 항목 생성:**
```markdown
## [버전] - 날짜

### 변경됨
- 인라인 스크립트를 `scripts/` 디렉토리로 추출
- 점진적 로딩 구현(67% 토큰 절약)
- 강력한 실행 지시문 추가

### 추가됨
- 포괄적인 문서가 있는 scripts/README.md
- SKILL.md의 단계 기반 참조 로딩

### 수정됨
- Claude가 이제 일관되게 미리 빌드된 스크립트 사용
- 토큰 소비량 ~3000에서 ~1000으로 감소

### 제거됨
- 문서의 인라인 코드 블록
```

## 일반적인 리팩토링 패턴

### 패턴 1: 큰 SKILL.md → 점진적 로딩

**이전:**
```
skill-name/
└── SKILL.md (2000줄)
```

**이후:**
```
skill-name/
├── SKILL.md (400줄)
└── references/
    ├── process/
    │   ├── phase1.md
    │   ├── phase2.md
    │   └── phase3.md
    └── support/
        └── advanced.md
```

**SKILL.md 변경:**
```markdown
## 참조 로딩 시기

### 단계 1: 초기 분석
- **로드**: [phase1.md](references/process/phase1.md)
- **언제**: 항상 여기서 시작

### 단계 2: 처리(조걶)
- **로드**: [phase2.md](references/process/phase2.md)

### 단계 3: 고급 기능(조걶)
- **로드**: [phase3.md](references/process/phase3.md)

**이전(문서에서):**

### 단계 3: 검증

````

**이후:**

**문서에서:**
````markdown
### 단계 3: 검증

**필수: 미리 빌드된 검증 스크립트 사용**

```bash
# EXECUTE_SCRIPT: scripts/validation/validate.sh
./scripts/validation/validate.sh < data.json
```

**중요:** 검증 로직을 인라인하지 마세요

**scripts/validation/validate.sh에서:**
```bash
#!/usr/bin/env bash
# validate.sh
# 목적: 데이터 형식과 콘텐츠 검증
# 사용법: ./validate.sh < data.json

set -e

# stdin에서 읽기
DATA=$(cat)

# 검증 로직
if ! echo "$DATA" | python3 -c "import json; json.loads(input())"; then
    echo "Invalid JSON" >&2
    exit 1
fi

# 추가 검증...
echo '{"status": "valid"}'
```

**scripts/README.md에서:**
```markdown
### validate.sh

**목적:** 데이터 형식과 콘텐츠 검증

**사용법:**
```bash
./scripts/validation/validate.sh < data.json
```

**입력:** stdin을 통한 JSON 데이터
**출력:** 성공 시 `{"status": "valid"}`
**종료 코드:**
- 0: 검증 통과
- 1: 검증 실패
```

## 영향 측정

### 이전/이후 지표

**토큰 사용량:**
```bash
# 대략적인 추정(단어 * 1.3)
cat git-commit.md | wc -w  # 이전: 1800단어 = ~2340토큰
cat git-commit.md | wc -w  # 이후: 400단어 = ~520토큰
# 절약: 78%
```

**파일 수:**
```bash
find skill-name -name "*.md" | wc -l
# 이전: 1 파일
# 이후: 6 파일(더 나은 조직)
```

**코드 블록:**

## 문제 해결

### 문제: Claude가 여전히 스크립트를 인라인함

**해결책:**
1. 지시문이 충분히 강력한지 확인(`필수`, `인라인 금지`)
2. `EXECUTE_SCRIPT:` 마커가 있는지 확인
3. 문서에서 남은 인라인 코드 확인
4. 금지 추가: "절대 명령어 재구성 금지"

### 문제: 참조가 로드되지 않음

**해결책:**
1. 상대 경로가 올바른지 확인
2. 파일명이 정확히 일치하는지 확인(대소문자 구분)
3. 링크를 수동으로 테스트
4. 올바른 마크다운 링크 구문 `[text](path)` 사용

### 문제: 토큰 사용량이 여전히 높음

**해결책:**
1. 더 많은 콘텐츠를 조걶 참조로 이동
2. 장황한 설명 제거
3. Claude가 기본을 알고 있다고 가정
4. 더 많은 코드를 스크립트로 추출
5. 스크립트가 실제로 사용되는지 확인(로드되지 않음)

### 문제: 스킬 동작이 변경됨

**해결책:**
1. 리팩토링 전후 출력 비교
2. 핵심 원칙이 변경되지 않았는지 확인
3. 모든 원래 기능이 여전히 작동하는지 확인
4. 원래 사용 사례 테스트
5. 복원하고 점진적 변경 시도

### 문제: 유지보수가 개선되지 않음

**해결책:**
1. 코드가 실제로 스크립트에 있는지 확인(문서에 없음)
2. 스크립트 조직이 논리적인지 확인
3. 문서가 스크립트를 명확하게 가리키는지 확인
4. 추가 통합 고려
