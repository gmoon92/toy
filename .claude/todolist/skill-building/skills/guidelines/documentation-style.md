# 스킬 문서화 스타일 가이드 (Documentation Style Guide for Skills)

Claude 4.x 프롬프팅 모범 사례를 기반으로 한 Claude Code 스킬 문서화 표준입니다.

## 핵심 원칙

### 1. 암시적이 아닌 명시적이기

Claude 4.x 모델은 명시적인 지시문에 뛰어납니다. 모호함을 피하세요.

**나쁜 예:**
```markdown
이 스킬은 커밋을 도와줍니다
```

**좋은 예:**
```markdown
Git 변경사항(staged + modified)을 분석하고, 일반적인 커밋 메시지를 생성하며,
Tidy First 원칙에 대해 검증하고, 승인 후 git commit을 실행합니다.
```

### 2. 무엇(what)이 아닌 왜(why) 설명하기

컨텍스트는 Claude가 목표를 이해하고 더 나은 결정을 내리는 데 도움이 됩니다.

**나쁜 예:**
```markdown
인라인 스크립트를 사용하지 마세요.
```

**좋은 예:**
```markdown
인라인 스크립트를 사용하지 마세요. 이유:
- 인라인 코드는 컨텍스트에 로드되어 토큰 낭비
- 미리 빌드된 스크립트는 결정적 실행 보장
- 스크립트 실행은 0토큰 비용(출력만 로드됨)
```

### 3. 보여주기, 그냥 말하지 않기

패턴을 보여주는 완전하고 현실적인 예시를 제공하세요.

**나쁜 예:**
```markdown
스크립트에 동사_명사 이름을 사용하세요.
```

**좋은 예:**
```markdown
## 스크립트 명명 규칙: 동사_명사

✅ `collect_changes.sh` - 명확한 동작과 대상
✅ `detect_scope.js` - 명시적인 목적
✅ `validate_message.py` - 명백한 행동

❌ `changes.sh` - 모호한 동작
❌ `scope.js` - scope에 대해 무엇을?
❌ `msg.py` - 불분명한 약어
```

## 구조 지침

### 프론트매터

간결하고 구체적으로 작성:

```yaml
---
name: skill-name
description: 정확한 목적과 동작을 설명하는 한 문장(도메인만이 아님)
disable-model-invocation: false
user-invocable: true
allowed-tools: Read, Write, Edit, Bash
---
```

**설명 규칙:**
- 스킬이 정확히 무엇을 하는지 명시(동사 + 결과)
- 주요 제약이나 원칙 포함
- 자동화나 검증이 관련되면 언급
- 가능하면 150자 미만 유지

### 개요 섹션

**구조:**
```markdown
## 개요

[스킬 목적에 대한 한 단락 설명]

**핵심 기능:**
- 결과와 함께 기능 1
- 결과와 함께 기능 2
- 결과와 함께 기능 3

**범위:**
- ✅ 스킬이 하는 것
- ✅ 스킬이 하는 것
- ❌ 절대: 스킬이 절대 하지 않는 것
- ❌ 절대: 스킬이 절대 하지 않는 것
```

**지침:**
- 가치 제안으로 시작
- 기능에는 글머리 목록 사용
- ✅/❌로 경계를 명시적으로 정의

### 참조 로딩 시기

토큰 절약을 위해 점진적 로딩을 설계하세요.

**구조:**
```markdown
## 참조 로딩 시기

### [단계 이름]
- **로드**: [relative/path/to/doc.md](path/to/doc.md)
- **로드**: [another/doc.md](path/to/doc.md)
- **목적**: 이것들이 가능하게 하는 것에 대한 간략한 설명
- **언제**: 이 단계를 트리거하는 조건

### [다른 단계]
- **로드**: [different/doc.md](path/to/doc.md)
- **목적**: 다른 기능
- **언제**: 다른 트리거
```

**지침:**
- 워크플로우 단계별로 참조 그룹화
- 각 참조가 왜 필요한지 설명
- 명확한 트리거 조건 명시
- 실제 파일 경로로 링크

## 실행 지시문 작성

### 강력한 키워드 사용

이들을 일관되게 적용하여 강조:

| 키워드 | 사용법 | 예시 |
|---------|-------|-------|
| `필수:` | 필수 동작 | `필수: 미리 빌드된 스크립트 실행` |
| `EXECUTE_SCRIPT:` | 스크립트 실행 마커 | `EXECUTE_SCRIPT: scripts/validate.sh` |
| `하지 마세요` | 명시적 금지 | `하지 마세요: 인라인 명령어` |
| `항상` | 강제 | `항상 스크립트 참조 사용` |
| `절대` | 강력한 금지 | `절대 코드 재구성 금지` |
| `중요:` | 높은 우선순위 | `중요: 먼저 git 상태 확인` |
| `주목:` | 중요한 참고사항 | `주목: 스크립트는 결정적임` |

### 지시문 패턴

스크립트 실행에 이 패턴 사용:

```markdown
**필수: 미리 빌드된 스크립트 사용 (인라인 명령어 금지)**

```bash
# EXECUTE_SCRIPT: scripts/category/script_name.sh

./scripts/category/script_name.sh < input.json > output.json
```

**중요:**
- 항상 `EXECUTE_SCRIPT:` 지시문을 통해 미리 빌드된 스크립트 사용
- 문서에서 명령어 재구성하지 마세요
- 인라인 bash 명령어 사용하지 마세요
- 스크립트는 결정적 실행 보장
```

**이것이 효과적인 이유:**
- `필수:` 기대 설정
- `EXECUTE_SCRIPT:` 정확한 스크립트 표시
- `하지 마세요` 명확한 금지 제공
- 글머리 기호가 요구사항 강조
- 컨텍스트가 왜 중요한지 설명

## 형식 제어

### 구조를 위한 XML 태그 사용

XML 태그는 Claude가 출력 구조를 이해하는 데 도움이 됩니다:

```markdown
**응답을 다음과 같이 구조화하세요:**

<analysis>
Git 변경사항에 대한 사실적 발견사항 제시:
- 수정된 파일 수
- Staged 변경사항 수
- 감지된 변경 유형(feature/fix/refactor)
</analysis>

<commit_message>
일반적인 커밋 메시지 생성:
type(scope): description

[선택적 본문]
</commit_message>

<validation>
원칙에 대해 검증:
- ✅ 논리적 독립성: yes/no
- ✅ Tidy First: yes/no
- ⚠️ 경고: 있을 경우
</validation>
```

**이점:**
- 명확한 출력 경계
- 일관된 구조
- 파싱하기 쉬움
- 자체 문서화

### 글머리 기호보다 산문 선호

설명형 콘텐츠의 경우 글머리 목록 대신 흐르는 단락을 사용하세요.

**나쁜 예:**
```markdown
스킬은 다음을 제공합니다:
- Git 변경사항 분석
- 커밋 메시지 생성
- 규칙 검증
- 자동 staging
```

**좋은 예:**
```markdown
스킬은 Git 변경사항을 분석하고, 프로젝트 규칙에 따라 커밋 메시지를 생성하며,
Tidy First 원칙에 대해 검증하고, 커밋 전 수정된 파일을 자동으로 staging합니다.
```

**글머리 기호를 사용해야 할 때:**
- 별개의 항목 나열(기능, 파일명, 단계)
- 명확한 경계 표시(✅/❌ 범위)
- 옵션이나 대안 제시
- 빠른 참조 섹션

## 코드 예시

### 완전하고 현실적인 예시

단편이 아닌 전체 컨텍스트를 보여주세요.

**나쁜 예:**
```markdown
detect_scope.js 사용:
```

**좋은 예:**
```markdown
## 범위 감지 예시

**EXECUTE_SCRIPT:** scripts/algorithms/detect_scope.js

```bash
# 입력: Git diff 출력
git diff --staged | ./scripts/algorithms/detect_scope.js

# 출력: 감지된 범위가 있는 JSON
{
  "scope": "auth",
  "confidence": 0.95,
  "files": ["src/auth/login.ts", "src/auth/session.ts"]
}
```

**작동 방식:**
스크립트는 파일 경로와 변경 패턴을 분석하여 변경의 논리적 범위를 결정합니다.
높은 신뢰도(>0.9)는 명확한 범위 경계를 나타냅니다.
```

### 입출력 형식 보여주기

스크립트의 경우 항상 I/O 문서화:

```markdown
## 스크립트: validate_message.sh

**목적:** 일반적인 커밋 형식에 대해 커밋 메시지 검증

**입력 형식:**
```text
type(scope): description

추가 컨텍스트를 제공하는 선택적 본문 단락.

참조가 있는 선택적 푸터.
```

**출력 형식:**
```json
{
  "valid": true,
  "errors": [],
  "warnings": ["Body exceeds 72 characters per line"]
}
```

**종료 코드:**
- 0: 유효한 메시지
- 1: 유효하지 않은 형식
- 2: 필수 필드 누락
```

## 정보 구성

### 단계 기반 조직

워크플로우 단계별로 문서 구성:

```markdown
## 단계 1: 분석
[분석 단계를 위한 문서]

## 단계 2: 생성
[생성 단계를 위한 문서]

## 단계 3: 검증
[검증 단계를 위한 문서]

## 단계 4: 실행
[실행 단계를 위한 문서]
```

**이점:**
- Claude의 워크플로우와 일치
- 점진적 로딩 가능
- 인지 부하 감소
- 명확한 경계

### 계층적 제목

일관된 제목 계층 사용:

```markdown
# 문서 제목 (H1) - 문서당 한 번

## 주요 섹션 (H2) - 주요 구분

### 하위 섹션 (H3) - 특정 주제

#### 세부사항 (H4) - 세부 내용

**굵게** - 단락 내 강조
```

## 작성 지침

### 능동태

명확성과 직접성을 위해 능동태 사용.

**나쁜 예 (수동태):**
```markdown
변경사항이 스킬에 의해 분석되고 커밋 메시지가 생성됩니다.
```

**좋은 예 (능동태):**
```markdown
스킬이 변경사항을 분석하고 커밋 메시지를 생성합니다.
```

### 현재시제

즉각성을 위해 현재시제로 작성.

**나쁜 예 (미래):**
```markdown
스크립트가 메시지 형식을 검증할 것입니다.
```

**좋은 예 (현재):**
```markdown
스크립트가 메시지 형식을 검증합니다.
```

### 지시문은 명령문으로

지시문에는 명령법을 사용.

**나쁜 예:**
```markdown
스크립트를 실행해야 합니다.
Claude는 미리 빌드된 스크립트를 사용해야 합니다.
```

**좋은 예:**
```markdown
스크립트를 실행하세요.
미리 빌드된 스크립트를 사용하세요.
항상 EXECUTE_SCRIPT 지시문을 통해 실행하세요.
```

## 피해야 할 안티패턴

### ❌ 모호한 설명

```markdown
## 나쁜 예
이것은 git 작업을 도와줍니다.

## 좋은 예
Git 변경사항을 분석하고 일반적인 커밋 메시지를 생성합니다.
```

### ❌ 컨텍스트 누락

```markdown
## 나쁜 예
인라인 코드를 사용하지 마세요.

## 좋은 예
인라인 코드를 사용하지 마세요. 토큰 낭비와 결정적 실행 방지 때문입니다.
미리 빌드된 스크립트는 0토큰 비용으로 일관성을 보장합니다.
```

### ❌ 불완전한 예시

```markdown
## 나쁜 예
스크립트 사용:
```bash
./script.sh
```

## 좋은 예
**EXECUTE_SCRIPT:** scripts/validate.sh

```bash
./scripts/validate.sh < message.txt > validation.json
```

**입력:** 일반 텍스트 커밋 메시지
**출력:** 오류 배열이 있는 JSON 검증 결과
```

### ❌ 모호한 범위

```markdown
## 나쁜 예
**범위:**
- 커밋 도움

## 좋은 예
**범위:**
- ✅ 일반적인 커밋 메시지 생성
- ✅ 프로젝트 규칙에 대해 검증
- ✅ 수정된 파일 자동 staging
- ❌ 절대: 커밋을 원격으로 푸시
- ❌ 절대: 커밋 기록 수정
```

### ❌ 마크다운 형식 과용

```markdown
## 나쁜 예
**이것**은 스킬이 *올바르게* 작동하기 위해 *정말* **중요**합니다!

## 좋은 예
이것은 스킬이 올바르게 작동하는 데 중요합니다.
```

**형식을 사용해야 할 때:**
- `**필수:**` - 지시문 키워드
- `**굵게**` - 구조화된 콘텐츠의 섹션 레이블
- `` `code` `` - 코드 참조, 파일명, 명령어
- 피하기: 과도한 굵게, 기울임, 또는 강조

## 문서 체크리스트

스킬 문서를 완성하기 전에:

- [ ] 프론트매터 설명이 명확하고 구체적임
- [ ] 개요가 가치와 범위를 명확히 설명
- [ ] 참조가 로드 조건과 함께 단계별로 구성됨
- [ ] 강력한 지시문 키워드가 일관되게 사용됨
- [ ] 스크립트 실행이 EXECUTE_SCRIPT 패턴 사용
- [ ] 예시가 입출력 형식과 함께 완전함
- [ ] 컨텍스트가 무엇(what)이 아닌 왜(why)를 설명
- [ ] 전체에 걸쳐 능동태와 현재시제 사용
- [ ] XML 태그가 복잡한 출력을 구조화
- [ ] 모호한 언어나 불분명한 지시문 없음
- [ ] 범위 경계가 ✅/❌로 명확히 정의됨
- [ ] 제목이 일관된 계층을 따름
- [ ] 코드 예시가 전체 컨텍스트를 보여줌
- [ ] 안티패턴이 피해짐

## 요약

효과적인 스킬 문서:
1. **명확한 지시문** - 정확히 무엇을 할지 알려주기
2. **풍부한 컨텍스트** - 왜 중요한지 설명
3. **완전한 예시** - 현실적인 사용법 보여주기
4. **강력한 지시문** - 키워드를 일관되게 사용
5. **구조화된 출력** - 명확성을 위한 XML 태그
6. **점진적 로딩** - 단계 기반 조직
7. **명확한 범위** - 경계를 명시적으로 정의
8. **토큰 효율성** - 스크립트 추출, 간결한 산문
