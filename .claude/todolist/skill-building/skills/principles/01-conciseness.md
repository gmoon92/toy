# 간결성: 핵심 원칙 (Conciseness: The Core Principle)

컨텍스트 윈도우는 모든 구성 요소가 공유합니다. 스킬을 간결하고 집중적으로 유지하세요.

## 간결성이 중요한 이유

컨텍스트 윈도우는 다음 간에 공유됩니다:
- 시스템 프롬프트
- 대화 기록
- 다른 스킬의 메타데이터
- 실제 요청

**핵심 인사이트:** 모든 토큰이 즉시 비용을 발생시키는 것은 아닙니다. 메타데이터(이름 + 설명)만 미리 로드됩니다. SKILL.md는 관련 있을 때 로드되며, 추가 파일은 필요할 때만 로드됩니다.

**그러나:** SKILL.md가 로드되면, 모든 토큰이 대화 기록 및 다른 컨텍스트와 경쟁합니다.

- "이 단락이 토큰 비용을 정당화하나?"

## 지침

### 1. 중복 설명 제거

❌ **명백한 개념 설명하지 마세요**
```markdown
데이터베이스는 구조화된 데이터를 저장하기 위한 시스템입니다...
```

✅ **도메인 지식 가정**
```markdown
BigQuery를 사용하여 sales 데이터베이스 쿼리.
```

### 2. 도입부 플러프 제거

❌ **장황한 도입부**
```markdown
이 섹션에서는 다양한 유형의 문서를 처리하기 위해 고려할 수 있는
다양한 접근 방식을 탐색해 보겠습니다...
```

✅ **직접적인 지시문**
```markdown
## 문서 처리

Word 파일: docx-js 사용
PDF: pdfplumber 사용
```

### 3. 보여주기, 과도하게 설명하지 않기

❌ **설명 중심**
```markdown
함수는 'path'라는 매개변수를 받는데, 이는 문서가 저장된 파일 시스템
위치를 나타냅니다. 이는 절대 경로나 상대 경로를 포함하는 문자열 값이어야
합니다...
```

✅ **코드 + 간략한 컨텍스트**
```markdown
```python
process_document(path="./docs/report.pdf")
```

`path`: 파일 위치 (절대 또는 상대)
```

### 4. Claude의 추론 신뢰

❌ **모든 것을 철자로 쓰기**
```markdown
파일이 존재하지 않으면 함수가 오류를 발생시킵니다. 함수를 호출하기 전에
먼저 파일이 존재하는지 확인하여 이 오류를 처리해야 합니다.
```

✅ **최소한의 안내**
```markdown
파일이 존재하지 않으면 `FileNotFoundError` 발생.
```

## 감사 질문

스킬 문서를 검토할 때, 다음을 물어보세요:

1. **Claude가 이것을 추론할 수 있나?**
   - 예라면: 제거하거나 크게 단축

2. **이 설명이 코드보다 긴가?**
   - 설명이 정말 필요한지 고려

3. **정보를 반복하고 있나?**
   - 통합 또는 교차 참조

4. **개발자에게 이것이 명백한가?**
   - 예라면: 제거

5. **이것이 가르치는 것인가 가이드하는 것인가?**
   - 스킬은 기본을 가르치지 않고 안내

## 토큰 예산 인식

### SKILL.md 구조

- **최적:** < 500줄
- **한계에 근접:** 참조 파일로 분할 시작
- **사용 패턴:** 점진적 공개 (structure/03-progressive-disclosure.md 참조)

### 콘텐츠 분할 시기

SKILL.md가 500줄에 근접하면:
1. 자체 포함된 섹션 식별
2. 참조 파일로 이동
3. SKILL.md에서 링크
4. 핵심 워크플로우는 SKILL.md에 유지

예시:
```markdown
## API 참조

SKILL.md의 핵심 엔드포인트

전체 참조: references/api-reference.md 참조
```
