# 과잉 엔지니어링 방지 (Avoid Overengineering)

스킬을 단순하고 집중적으로 유지하기 위한 지침입니다.

## 문제

Claude Opus 4.5는 다음을 할 수 있습니다:
- 불필요한 추상화 생성
- 요청받지 않은 기능 추가
- 일회성 작업을 위한 유틸리티 빌드
- 가상의 미래 요구사항을 위한 설계
- 과도한 오류 처리 추가

## 해결책

과잉 엔지니어링을 명시적으로 금지하고 최소 범위를 정의하세요.

---

## 패턴

```markdown
<avoid_overengineering>
직접 요청되었거나 명확히 필요한 변경사항만 수행하세요.
해결책을 단순하고 집중적으로 유지하세요.

하지 마세요:
- 요구사항을 넘어선 기능 추가
- 관련 없는 코드 리팩토링
- 일회성 작업을 위한 유틸리티 생성
- 가상의 미래 요구사항을 위한 설계
- 불가능한 시나리오에 대한 오류 처리 추가
- 필요하지 않은 구성 시스템 빌드

하세요:
- 특정 문제 해결
- 기존 추상화 사용
- 낶부 코드와 프레임워크 신뢰
- 시스템 경계에서만 검증
- 현재 작업을 위한 최소 코드 작성
</avoid_overengineering>
```

---

## 예시: 스크립트 생성

```markdown
## 스크립트 범위

<avoid_overengineering>
스크립트를 생성할 때:

**필수:**
- ✅ 명시된 목적을 위한 핵심 기능
- ✅ 지정된 대로 입력/출력
- ✅ 외부 입력에 대한 오류 처리

**필수 아님:**
- ❌ 간단한 스크립트를 위한 구성 파일
- ❌ 디버그 출력을 위한 로깅 프레임워크
- ❌ 단일 구현을 위한 추상 기본 클래스
- ❌ 고정 인수를 위한 명령줄 파서
- ❌ 한 번만 사용되는 도우미 유틸리티

스크립트를 단순하고 집중적이며 유지보수 가능하게 유지하세요.
</avoid_overengineering>
```

## 상세 지침

```markdown
<avoid_overengineering_detailed>
과잉 엔지니어링을 피하세요. 직접 요청되었거나 명확히 필요한 변경사항만 수행하세요.
해결책을 단순하고 집중적으로 유지하세요.

요청된 것 이상의 기능, 코드 리팩토링, 또는 "개선"을 추가하지 마세요.
버그 수정은 주변 코드 정리가 필요하지 않습니다.
간단한 기능은 추가 구성 가능성이 필요하지 않습니다.
변경하지 않은 코드에 문서 문자열, 주석, 또는 타입 어노테이션을 추가하지 마세요.
논리가 자명하지 않은 경우에만 주석을 추가하세요.

발생할 수 없는 시나리오에 대한 오류 처리, 폴백, 또는 검증을 추가하지 마세요.
낶부 코드와 프레임워크 보장을 신뢰하세요.
시스템 경계(사용자 입력, 외부 API)에서만 검증하세요.
코드를 변경할 수 있을 때 기능 플래그나 하위 호환성 셉을 사용하지 마세요.

일회성 작업을 위한 도우미, 유틸리티, 또는 추상화를 생성하지 마세요.
가상의 미래 요구사항을 위한 설계를 하지 마세요.
적절한 복잡성 수준은 현재 작업에 필요한 최소한의 것입니다—
세 개의 유사한 코드 줄은 조기 추상화보다 낫습니다.

가능할 때 기존 추상화를 재사용하고 DRY 원칙을 따르세요.
</avoid_overengineering_detailed>
```

## 범위 정의

범위 안에 무엇이 있고 밖에 무엇이 있는지 명확히 정의하세요:

```markdown
## 범위

**하세요:**
- ✅ 인라인 코드를 스크립트로 추출 (요청됨)
- ✅ 실행 지시문 추가 (필요함)
- ✅ 문서 업데이트 (필요함)

**하지 마세요:**
- ❌ 불가능한 시나리오에 대한 오류 처리 추가
- ❌ 일회성 작업을 위한 유틸리티 생성
- ❌ 가상의 미래 요구사항을 위한 설계
- ❌ 명시적으로 필요하지 않은 경우 구성 시스템 추가
```

## 테스트 집중

테스트에 과적합되지 않도록 방지하세요:

```markdown
사용 가능한 표준 도구를 사용하여 고품질의 범용 솔루션을 작성하세요.
작업을 더 효율적으로 수행하기 위해 도우미 스크립트나 우회 방법을 생성하지 마세요.
테스트 케이스뿐만 아니라 모든 유효한 입력에 대해 올바르게 작동하는 솔루션을 구현하세요.
특정 테스트 입력에 대해서만 작동하는 값을 하드코딩하거나 솔루션을 생성하지 마세요.
대신, 문제를 일반적으로 해결하는 실제 로직을 구현하세요.

문제 요구사항을 이해하고 올바른 알고리즘을 구현하는 데 집중하세요.
테스트는 정확성을 검증하기 위한 것이지 솔루션을 정의하기 위한 것이 아닙니다.
모범 사례와 소프트웨어 설계 원칙을 따르는 원칙에 기반한 구현을 제공하세요.

작업이 불합리하거나, 실행 불가능하거나, 또는 테스트가 잘못된 경우,
우회하려고 시도하는 대신 저에게 알려주세요. 솔루션은 견고하고,
유지보수 가능하며, 확장 가능해야 합니다.
```

## 하위 호환성

불필요한 하위 호환성 해킹을 피하세요:

```markdown
다음과 같은 하위 호환성 해킹을 피하세요:
- 사용하지 않는 _vars 이름 바꾸기
- 타입 재낸스포트
- 제거된 코드에 // removed 주석 추가

사용하지 않는 것이 확실하면 완전히 삭제하세요.
```

## 요약

다음을 통해 스킬을 단순하게 유지하세요:
- **명확한 범위** - 명확한 경계 정의
- **최소 코드** - 필요한 것만
- **프레임워크 신뢰** - 모든 것을 검증하지 마세요
- **미래 대비 없음** - 현재 문제 해결
- **사용하지 않는 것 삭제** - 하위 호환성 해킹 없음
- **표준 도구** - 우회 방법 생성하지 마세요
