## 치명 문제 재검증 결과

### [치명 1] 파일 참조 버그 — **해결됨**

**검증 결과**: 해결됨.

Round 3는 `See: assets/templates/{archetype}.md` 방식을 완전히 폐기하고 "옵션 C"(SKILL.md Task prompt에 성향별 구조 직접 내장)를 선택했다. 6개 성향의 시스템 프롬프트 구조, 출력 형식, 핵심 패턴이 Task prompt 블록 내에 마크다운 코드 블록으로 직접 작성되어 있다.

**근거**: Task 에이전트가 실행되는 시점에 SKILL.md의 전체 내용(성향별 구조 포함)이 이미 메인 컨텍스트로 전달된 상태이므로, 파일을 별도로 Read할 필요 없이 6개 성향 블록 모두 Task prompt에 포함되어 있다. 파일 참조 실패 가능성이 구조적으로 제거되었다.

**잔여 위험**: 없음.

### [치명 2] Archetype 추론 — **해결됨 (단, 잠재적 추론 실패 케이스 존재)**

**검증 결과**: 설계 수준에서 해결됨. 그러나 추론 실패 케이스는 완전히 차단되지 않는다.

**해결된 부분**:
- `{archetype}` 자리표시자가 제거되고 서술형 지시로 대체됨 (치명 2A 해결).
- 성향이 사용자 입력 항목에서 제거되고 Claude의 내부 추론으로 처리됨 (치명 2B 해결).
- 성향 추론 규칙이 키워드 매핑 표로 SKILL.md에 명시됨.

**잠재적 추론 실패 케이스**:

1. **복합 목적 시 분류 모호**: "코드 리뷰 후 자동으로 버그를 수정해주는 에이전트"는 `reviewer`와 `fixer` 두 성향에 동시에 해당한다. 키워드 우선순위 규칙이 없으므로 Claude가 어느 성향을 선택할지 비결정적이다.

2. **기본값 `executor` 적용 범위 과도**: 분류 불가 시 `executor`를 기본값으로 지정했으나, `executor`는 전체 도구 상속(`tools` 필드 생략)을 사용한다. 분류 불가 에이전트에 전체 권한을 부여하는 것은 최소 권한 원칙 위반이다. 예를 들어 "회의록을 정리해주는 에이전트"는 분류 불가일 수 있는데, 이에 `executor`(전체 권한)를 할당하면 적절하지 않다.

3. **키워드 미매핑 언어**: 한국어 키워드만 표에 있다. 사용자가 영어로 요청할 경우("Create an agent that reviews pull requests") 키워드 매핑이 동작하지 않을 수 있다. Claude의 자연어 이해 능력에 의존하는 구조로, 표 자체는 완전한 기준이 아니다.

### [치명 3] 이중 소스(examples.md + templates) — **부분 해결**

**검증 결과**: 구조적 공존 문제는 해결됐으나, examples.md의 Read 지시 메커니즘에 미결 문제가 있다.

**해결된 부분**: templates/ 디렉토리를 생성하지 않기로 결정함으로써 이중 소스 공존 문제를 근본적으로 제거했다. 성향별 구조는 SKILL.md에 내장되고, examples.md는 "완성된 예시"로 단일 역할을 유지한다.

**미결 문제 — Read 지시의 실행 보장 불명확**:

Round 3 Task prompt 내 지시:
```
[예시 참조]
완성된 에이전트 예시를 참고하려면 다음 파일을 Read 도구로 읽으세요:
- 절대 경로: {skill_root}/references/examples.md
```

이 지시에는 두 가지 문제가 있다:

1. **`{skill_root}`는 자동 치환되지 않는다**: Round 3가 치명 2A에서 해결한 것과 동일한 자리표시자 문제가 examples.md 참조에서 재발했다. `{skill_root}`의 실제 값(예: `/Users/gmoon/.claude/skills/subagent-creator`)은 메인 에이전트가 알고 있지만, Task 에이전트에게 실제 경로 문자열로 치환해서 전달해야 한다는 지시가 SKILL.md 워크플로우에 없다.

2. **Read 지시가 선택적(optional)이다**: "참고하려면 읽으세요"는 필수 지시가 아니라 권고다. Task 에이전트가 examples.md를 읽지 않고도 파일을 생성할 수 있으며, 이 경우 examples 없이 성향 구조만 참조하게 된다. 이것이 의도적 설계인지, 누락인지 불명확하다.

## 새로 발견된 문제

### [신규 1] SKILL.md 과부하 — 스킬 가이드와 템플릿 혼재

**문제**: Round 3의 확정된 SKILL.md는 다음 역할을 모두 한 파일에 담는다:
- 스킬 메타 정보 (frontmatter description)
- 서브 에이전트 파일 형식 안내
- 성향 분류표 (추론 기준)
- 워크플로우 절차 (Step 1-3)
- 6개 성향의 시스템 프롬프트 구조 + 출력 형식 (실질적 템플릿)
- References 섹션

블루팀 추정("300-400줄 수준으로 관리 가능")과 달리, 현재 Task prompt 블록 내 6개 성향 구조만 이미 약 130줄이고(Round 3 문서의 라인 193-302), 워크플로우 전체 포함 시 최소 200줄 이상이다. 여기에 frontmatter, 파일 형식 안내, 성향 분류표, References까지 더하면 실제 SKILL.md는 250-300줄이 된다.

**위험**: 길이 자체보다, "스킬 사용 가이드"와 "에이전트 생성 템플릿 소재"가 같은 파일에 혼재하여 유지보수 시 어느 섹션을 수정해야 하는지 불명확해진다. 성향 구조를 수정할 때마다 Task prompt의 긴 블록에서 원하는 성향을 찾아야 한다.

**영향도**: 중간 (기능상 문제 없음, 유지보수 복잡성 증가).

---

### [신규 2] References 섹션 주석과 치명 3 해결 방식의 논리적 충돌

**문제**: Round 3의 References 섹션:
```
- `references/examples.md` — 완성된 서브 에이전트 예시 (Task 에이전트가 Read로 직접 로드)
```

이 주석은 "Task 에이전트가 Read로 직접 로드"한다고 명시한다. 그런데 현재 실제 SKILL.md(파일 시스템)의 References 섹션에는 이와 반대되는 주의사항이 있다:
```
> **주의**: 이 파일들은 메인 컨텍스트에서 직접 로드하지 마세요. Task 에이전트에 참조로만 전달하세요.
```

Round 3 설계가 "Task 에이전트가 Read로 직접 로드"하도록 바꾸었다면, 이 기존 주의사항은 삭제 또는 수정되어야 한다. Round 3 문서에는 이 충돌을 해소하는 언급이 없다.

**실질적 영향**: 구현자가 Round 3 설계를 실제 파일에 적용할 때 References 섹션 주의사항을 어떻게 처리해야 하는지 명세가 없다.

---

### [신규 3] 성향 분류표의 executor tools 불완전 명세

**문제**: 성향 분류표에서 executor의 tools 항목:
```
| executor | 실행, 배포, 테스트, 빌드 | (생략 — 전체 상속) | inherit | default |
```

"전체 상속"을 `tools` 필드 생략으로 표현했는데, Task prompt 내 성향별 구조 가이드에서 executor는 별도 `tools` 항목 없이 frontmatter를 작성하게 된다. 그러나 "파일 생성 규칙"에는:
```
frontmatter에 name, description, tools(executor 제외), model, permissionMode(default 제외) 포함
```
이라고 명시되어 있다. `executor 제외`가 명시적으로 있으므로 이 부분은 일관성이 있다.

단, executor에게 전체 도구 권한을 기본값으로 부여하는 것은 보안 관점에서 설명이 필요한 결정이다. 이 판단 근거가 어디에도 기술되어 있지 않다.

---

### [신규 4] `{skill_root}` 자리표시자 재등장 (치명 2A의 재발)

Round 3는 치명 2A에서 `{archetype}`과 같은 자리표시자를 제거하겠다고 선언했다. 그러나 Task prompt 내 examples.md 참조에서 `{skill_root}`라는 새로운 자리표시자를 도입했다:

```
- 절대 경로: {skill_root}/references/examples.md
```

이것은 치명 2A가 해결하려 했던 동일한 문제의 재발이다. 치명 2A 해결 원칙("서술형 지시로 대체, 실행 시 실제 값으로 치환")을 이 자리표시자에도 동일하게 적용해야 한다. 현재 설계에는 이 변환 책임이 어디에 있는지 명시되어 있지 않다.

## 최종 판정

**조건부 합격**

**근거**:

Round 3는 Round 2의 3개 치명 문제에 대해 올바른 방향으로 해결했다. 특히 치명 1(외부 파일 참조 완전 제거)과 치명 2B(성향을 사용자 입력에서 내부 추론으로 전환)는 설계 수준에서 완결되었다.

그러나 다음 두 가지 조건이 충족되지 않으면 실제 구현 시 동일한 버그가 재발한다:

**조건 1 (필수)**: `{skill_root}` 자리표시자를 제거해야 한다. 메인 에이전트가 스킬 루트 경로를 알고 있다면 Task prompt에 실제 절대 경로로 삽입해야 하며, 알 수 없다면 examples.md 참조를 선택적이 아닌 명시적 Read 지시로 바꾸거나 완전히 제거해야 한다.

**조건 2 (필수)**: 복합 목적(성향 복수 해당) 케이스에 대한 우선순위 규칙이 필요하다. 최소한 "성향이 복수 해당할 경우 더 제한적인 권한을 가진 성향을 선택한다" 또는 "더 구체적인 키워드를 우선한다"는 규칙을 명시해야 한다.

나머지 신규 문제들(SKILL.md 혼재, executor 전체 권한 근거 미기술)은 기능 동작에는 영향이 없으므로 Round 5에서 선택적으로 개선 가능하다.

## Round 5를 위한 조건/권고사항

### 필수 반영 (미반영 시 구현 버그 발생)

1. **`{skill_root}` 자리표시자 제거**: Task prompt 내 examples.md 절대 경로를 두 가지 중 하나로 처리하라.
   - 방법 A: 워크플로우 Step 3에서 메인 에이전트가 실제 경로를 계산하여 Task prompt에 문자열로 삽입한다는 지시 추가.
   - 방법 B: "참고하려면 읽으세요" 대신 "반드시 다음 절대 경로를 Read 도구로 읽으세요: `/절대경로/references/examples.md`"로 경로를 하드코딩하거나, examples.md 내용 자체를 SKILL.md 내 별도 섹션으로 이전한다.

2. **복합 성향 우선순위 규칙 추가**: 성향 추론 규칙 표에 "키워드가 복수 성향에 해당할 경우 우선순위: auditor > reviewer > fixer > analyst > writer > executor" 또는 유사한 명시적 규칙 추가.

### 권고 반영 (품질 개선)

3. **executor 기본값 전체 권한 근거 명시**: 성향 분류표 또는 파일 생성 규칙에 "executor는 실행 작업 특성상 필요한 도구 범위를 사전에 한정하기 어려워 전체 상속을 기본값으로 함"과 같은 한 줄 주석 추가.

4. **SKILL.md References 섹션 기존 주의사항 삭제 또는 업데이트**: "메인 컨텍스트에서 직접 로드하지 마세요" 주의사항이 Round 3 설계(Task 에이전트가 Read로 로드)와 충돌하므로 제거 또는 수정.

5. **영어 키워드 추론 안내 추가**: 성향 추론 규칙이 한국어 키워드 기반임을 명시하고, 영어 요청 시 의미적 유사도로 판단한다는 보완 지시 추가.
