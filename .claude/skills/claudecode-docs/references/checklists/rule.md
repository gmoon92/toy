## 1. 룰 특화 파일 위치 검증

- [ ] Rule 파일이 `.claude/rules/` 디렉토리에 위치하는지
- [ ] 파일 확장자가 `.md`인지
- [ ] 파일명이 규칙의 목적을 명확히 설명하는지 (예: `code-style.md`, `api-design.md`)
- [ ] 하위 디렉토리를 사용하여 규칙을 주제별로 그룹화했는지 (선택사항, 예: `frontend/`, `backend/`)

## 2. 룰 특화 프론트매터 검증

### 선택 필드 유효성

- [ ] `paths` 필드가 문자열 배열(`string[]`) 또는 단일 문자열(`string`)인지
- [ ] `paths`의 각 패턴이 유효한 Glob 패턴인지
- [ ] `paths` 패턴에 오타나 문법 오류가 없는지 (예: `**` 사용, `{}` 확장 등)
- [ ] `description` 필드가 문자열인지
- [ ] `description`이 200자 이하인지 (간결한 설명 권장)
- [ ] `description`이 규칙의 목적을 명확히 설명하는지

### Glob 패턴 검증

- [ ] `**/*.ts` - 모든 디렉토리의 TypeScript 파일
- [ ] `src/**/*` - `src/` 하위 모든 파일
- [ ] `*.{ts,tsx}` - 다중 확장자 패턴
- [ ] `src/components/*.tsx` - 특정 디렉토리의 파일
- [ ] 패턴에 공백이나 특수문자가 적절히 처리되었는지

---

## 3. 룰 특화 본문 내용 검증

### 규칙의 명확성

- [ ] 각 규칙이 구체적이고 실행 가능한지
- [ ] 모호한 표현("적절히", "가능하면", "잘") 대신 구체적인 기준이 있는지
- [ ] 규칙이 측정 가능한 기준을 포함하는지 (예: "2-space indentation", "max 80 chars")

### 규칙의 구조

- [ ] 마크다운 헤더를 사용하여 규칙을 그룹화했는지
- [ ] 관련 규칙이 섹션별로 묶여 있는지
- [ ] 중요도 순서로 규칙이 배치되었는지 (필수 → 권장 → 선택)

### 예시 포함 여부

- [ ] 복잡한 규칙에 대해 코드 예시가 포함되었는지
- [ ] "DO"와 "DON'T" 예시가 명확히 구분되었는지 (선택사항)
- [ ] 예시가 실제 프로젝트 코드와 일치하는지

---

## 4. 규칙 충돌 방지 검증

### 다른 Rule과의 관계

- [ ] 동일한 `paths` 패턴을 가진 Rule이 없는지 (중복 가능성 확인)
- [ ] 서로 상충되는 규칙이 없는지 (예: A규칙은 2-space, B규칙은 4-space)
- [ ] 더 구체적인 패턴의 Rule이 더 일반적인 Rule보다 우선순위가 명확한지

### CLAUDE.md와의 관계

- [ ] Rule이 CLAUDE.md의 내용과 충돌하지 않는지
- [ ] CLAUDE.md의 일반 지침을 Rule이 구체화하는 형태인지
- [ ] 중복된 내용이 CLAUDE.md와 Rule 사이에 없는지

---

## 5. 실용성 검증

### 적용 가능성

- [ ] 규칙이 프로젝트의 실제 상황에 적용 가능한지
- [ ] 규칙이 너무 엄격하거나 느슨하지 않은지
- [ ] 팀이 규칙을 일관되게 따를 수 있는지

### 유지보수성

- [ ] 규칙이 변경에 유연한지 (기술 스택 변경 시 규칙도 함께 업데이트 필요)
- [ ] 규칙이 너무 많지 않은지 (200줄 이하 권장)
- [ ] 규칙이 명확한 근거나 문서 링크를 포함하는지 (선택사항)

---

## 6. 예시 기반 검증

### Path-specific Rule 예시

```markdown
---
paths:
  - "src/api/**/*.ts"
description: API 개발 규칙
---

# API Development Rules

## 입력 검증
- 모든 API 엔드포인트는 입력 데이터 검증을 포함해야 함
- zod 또는 class-validator 사용 권장

## 에러 응답
- 표준 에러 응답 형식 사용
- HTTP 상태 코드 적절히 사용
```

### 일반 Rule 예시

```markdown
# Code Style Rules

## 들여쓰기
- 2-space 들여쓰기 사용
- 탭 대신 스페이스 사용

## 명명 규칙
- 변수: camelCase
- 상수: UPPER_SNAKE_CASE
- 클래스: PascalCase
```
