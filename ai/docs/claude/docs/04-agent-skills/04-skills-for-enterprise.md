# [엔터프라이즈용 스킬](https://platform.claude.com/docs/en/agents-and-tools/agent-skills/enterprise)

엔터프라이즈 규모에서 Agent Skills를 배포하기 위한 거버넌스, 보안 검토, 평가 및 조직 가이드입니다.

---

이 가이드는 조직 전체에서 Agent Skills를 관리해야 하는 엔터프라이즈 관리자 및 아키텍트를 위한 것입니다. 
대규모로 스킬을 검증하고, 평가하고, 배포하고, 관리하는 방법을 다룹니다. 
작성 가이드는 [모범 사례](../04-agent-skills/03-best-practices.md)를 참조하세요. 아키텍처 세부 정보는 [스킬 개요](../04-agent-skills/01-overview.md)를 참조하세요.

## 보안 검토 및 검증

엔터프라이즈에서 스킬을 배포하려면 두 가지 뚜렷한 질문에 답해야 합니다:

1. **스킬이 일반적으로 안전한가?** 플랫폼 수준의 보안 세부 정보는 개요의 [보안 고려사항](../04-agent-skills/01-overview.md) 섹션을 참조하세요.
2. **특정 스킬을 어떻게 검증하는가?** 아래의 위험 평가 및 검토 체크리스트를 사용하세요.

### 위험 등급 평가

배포를 승인하기 전에 각 스킬을 다음 위험 지표에 대해 평가하세요:

| 위험 지표         | 확인 사항                                                         | 우려 수준                          |
|---------------|---------------------------------------------------------------|--------------------------------|
| 코드 실행         | 스킬 디렉터리의 스크립트(`*.py`, `*.sh`, `*.js`)                         | 높음: 스크립트가 전체 환경 액세스 권한으로 실행됨   |
| 명령 조작         | 안전 규칙을 무시하거나, 사용자로부터 작업을 숨기거나, Claude의 동작을 조건부로 변경하도록 지시하는 내용 | 높음: 보안 제어를 우회할 수 있음            |
| MCP 서버 참조     | MCP 도구를 참조하는 명령(`ServerName:tool_name`)                       | 높음: 스킬 자체를 넘어 액세스 확장           |
| 네트워크 액세스 패턴   | URL, API 엔드포인트, `fetch`, `curl` 또는 `requests` 호출              | 높음: 잠재적인 데이터 유출 경로             |
| 하드코딩된 자격 증명   | 스킬 파일 또는 스크립트의 API 키, 토큰 또는 비밀번호                              | 높음: Git 히스토리 및 컨텍스트 윈도우에 비밀 노출 |
| 파일 시스템 액세스 범위 | 스킬 디렉터리 외부 경로, 광범위한 glob 패턴, 경로 순회(`../`)                     | 중간: 의도하지 않은 데이터에 액세스할 수 있음     |
| 도구 호출         | Claude에게 bash, 파일 작업 또는 기타 도구를 사용하도록 지시하는 명령                  | 중간: 수행되는 작업 검토 필요              |

### 검토 체크리스트

타사 또는 내부 기여자의 스킬을 배포하기 전에 다음 단계를 완료하세요:

1. **모든 스킬 디렉터리 콘텐츠를 읽으세요.** 
   - SKILL.md, 참조된 모든 마크다운 파일, 번들된 스크립트 또는 리소스를 검토하세요.
2. **스크립트 동작이 명시된 목적과 일치하는지 확인하세요.** 
   - 샌드박스 환경에서 스크립트를 실행하고 출력이 스킬의 설명과 일치하는지 확인하세요.
3. **적대적 명령을 확인하세요.** 
   - Claude에게 안전 규칙을 무시하거나, 사용자로부터 작업을 숨기거나, 응답을 통해 데이터를 유출하거나, 특정 입력에 따라 동작을 변경하도록 지시하는 내용을 찾으세요.
4. **외부 URL 가져오기 또는 네트워크 호출을 확인하세요.** 
   - 스크립트와 명령에서 네트워크 액세스 패턴(`http`, `requests.get`, `urllib`, `curl`, `fetch`)을 검색하세요.
5. **하드코딩된 자격 증명이 없는지 확인하세요.** 
   - 스킬 파일에서 API 키, 토큰 또는 비밀번호를 확인하세요. 
   - 자격 증명은 환경 변수나 안전한 자격 증명 저장소를 사용해야 하며, 스킬 콘텐츠에 절대 나타나서는 안 됩니다.
6. **스킬이 Claude에게 호출하도록 지시하는 도구와 명령을 식별하세요.** 
   - 모든 bash 명령, 파일 작업 및 도구 참조를 나열하세요. 
   - 스킬이 파일 읽기와 네트워크 도구를 함께 사용할 때 결합된 위험을 고려하세요.
7. **리디렉션 대상을 확인하세요.** 
   - 스킬이 외부 URL을 참조하는 경우 예상 도메인을 가리키는지 확인하세요.
8. **데이터 유출 패턴이 없는지 확인하세요.** 
   - 민감한 데이터를 읽은 다음 외부 전송을 위해 작성, 전송 또는 인코딩하는 명령을 찾으세요. 
   - Claude의 대화 응답을 통한 유출도 포함됩니다.

> 신뢰할 수 없는 소스의 스킬을 전체 감사 없이 배포하지 마세요. 
> 악의적인 스킬은 Claude가 임의의 코드를 실행하거나, 민감한 파일에 액세스하거나, 외부로 데이터를 전송하도록 지시할 수 있습니다. 
> 스킬 설치를 프로덕션 시스템에 소프트웨어를 설치하는 것과 동일한 엄격함으로 취급하세요.

## 배포 전 스킬 평가

스킬이 잘못 트리거되거나, 다른 스킬과 충돌하거나, 부적절한 지침을 제공하면 에이전트 성능이 저하될 수 있습니다. 
프로덕션 배포 전에 평가를 요구하세요.

### 평가 대상

스킬을 배포하기 전에 다음 차원에 대한 승인 게이트를 수립하세요:

| 차원      | 측정 대상                                              | 실패 예시                                            |
|---------|----------------------------------------------------|--------------------------------------------------|
| 트리거 정확도 | 스킬이 올바른 쿼리에 대해 활성화되고 관련 없는 쿼리에 대해서는 비활성 상태를 유지하는가? | 사용자가 데이터에 대해 논의하고 싶을 때도 스프레드시트가 언급될 때마다 스킬이 트리거됨 |
| 격리 동작   | 스킬이 단독으로 올바르게 작동하는가?                               | 스킬이 디렉터리에 존재하지 않는 파일을 참조함                        |
| 공존성     | 이 스킬을 추가하면 다른 스킬이 저하되는가?                           | 새 스킬의 설명이 너무 광범위하여 기존 스킬의 트리거를 가로챔               |
| 명령 준수   | Claude가 스킬의 명령을 정확하게 따르는가?                         | Claude가 검증 단계를 건너뛰거나 잘못된 라이브러리를 사용함              |
| 출력 품질   | 스킬이 정확하고 유용한 결과를 생성하는가?                            | 생성된 보고서에 서식 오류나 누락된 데이터가 있음                      |

### 평가 요구사항

스킬 작성자에게 스킬당 3-5개의 대표 쿼리가 포함된 평가 스위트를 제출하도록 요구하세요.
스킬이 트리거되어야 하는 경우, 트리거되지 않아야 하는 경우, 애매한 엣지 케이스를 포함해야 합니다. 
조직에서 사용하는 모델(Haiku, Sonnet, Opus)에 걸쳐 테스트를 요구하세요. 
스킬 효과는 모델마다 다르기 때문입니다.

평가 구축에 대한 자세한 가이드는 모범 사례의 [평가 및 반복](../04-agent-skills/03-best-practices.md)을 참조하세요. 
일반적인 평가 방법론은 [테스트 케이스 개발](../09-test-evaluate/02-develop-tests.md)을 참조하세요.

### 라이프사이클 결정을 위한 평가 활용

평가 결과는 언제 조치를 취해야 하는지 알려줍니다:

- **트리거 정확도 감소:** 스킬의 설명 또는 명령 업데이트
- **공존성 충돌:** 중복되는 스킬 통합 또는 설명 범위 좁히기
- **지속적으로 낮은 출력 품질:** 명령 재작성 또는 검증 단계 추가
- **업데이트에도 지속적인 실패:** 스킬 폐기

## 스킬 라이프사이클 관리

- **계획**
  - 반복적이거나, 오류가 발생하기 쉽거나, 전문 지식이 필요한 워크플로를 식별하세요. 
  - 이를 조직 역할에 매핑하고 스킬 후보를 결정하세요.
- **생성 및 검토**
  - 스킬 작성자가 [모범 사례](../04-agent-skills/03-best-practices.md)를 따르도록 하세요. 위의 [검토 체크리스트](#검토-체크리스트)를 사용하여 보안 검토를 요구하세요. 
  - 승인 전에 평가 스위트를 요구하세요. 
  - 직무 분리를 수립하세요: 스킬 작성자는 자신의 검토자가 되어서는 안 됩니다.
- **테스트**
  - 격리(스킬 단독) 및 기존 스킬과 함께(공존성 테스트) 평가를 요구하세요. 
  - 프로덕션 승인 전에 활성 스킬 세트 전체에서 트리거 정확도, 출력 품질 및 회귀 부재를 확인하세요.
- **배포**
  - 워크스페이스 전체 액세스를 위해 Skills API를 통해 업로드하세요. 
  - 업로드 및 버전 관리는 [API로 스킬 사용](../04-agent-skills/05-using-skills-with-api.md)을 참조하세요. 
  - 내부 레지스트리에 목적, 소유자 및 버전과 함께 스킬을 문서화하세요.
- **모니터링**
  - 사용 패턴을 추적하고 사용자로부터 피드백을 수집하세요. 
  - 워크플로와 모델이 발전함에 따라 드리프트나 회귀를 감지하기 위해 정기적으로 평가를 다시 실행하세요. 
  - 사용 분석은 현재 Skills API를 통해 제공되지 않습니다. 
  - 요청에 포함된 스킬을 추적하기 위해 애플리케이션 수준 로깅을 구현하세요.
- **반복 또는 폐기**
  - 새 버전을 승격하기 전에 전체 평가 스위트를 통과하도록 요구하세요. 
  - 워크플로가 변경되거나 평가 점수가 감소할 때 스킬을 업데이트하세요. 
  - 평가가 지속적으로 실패하거나 워크플로가 폐기될 때 스킬을 폐기하세요.

## 대규모 스킬 조직화

### 회상 제한

일반적인 가이드라인으로, 신뢰할 수 있는 회상 정확도를 유지하기 위해 동시에 로드되는 스킬 수를 제한하세요. 
각 스킬의 메타데이터(이름 및 설명)는 시스템 프롬프트에서 주목을 받기 위해 경쟁합니다. 
너무 많은 스킬이 활성화되면 Claude가 올바른 스킬을 선택하지 못하거나 관련 스킬을 완전히 놓칠 수 있습니다. 
평가 스위트를 사용하여 스킬을 추가할 때 회상 정확도를 측정하고, 성능이 저하되면 추가를 중지하세요.

API 요청은 요청당 최대 8개의 스킬을 지원합니다([API로 스킬 사용](../04-agent-skills/05-using-skills-with-api.md) 참조). 
역할에 단일 요청이 지원하는 것보다 더 많은 스킬이 필요한 경우, 좁은 스킬을 더 광범위한 스킬로 통합하거나 작업 유형에 따라 요청을 다른 스킬 세트로 라우팅하는 것을 고려하세요.

### 구체적으로 시작하고 나중에 통합

팀이 광범위하고 다목적 스킬보다는 좁고 워크플로별 스킬로 시작하도록 권장하세요. 
조직 전체에서 패턴이 나타나면 관련 스킬을 역할 기반 번들로 통합하세요.

> 평가를 사용하여 언제 통합할지 결정하세요. 통합된 스킬의 평가가 대체하는 개별 스킬과 동등한 성능을 확인할 때만 좁은 스킬을 더 광범위한 스킬로 병합하세요.

**진행 예시**:

- 시작: `formatting-sales-reports`, `querying-pipeline-data`, `updating-crm-records`
- 통합: `sales-operations` (평가가 동등한 성능을 확인할 때)

### 명명 및 카탈로그화

조직 전체에서 일관된 명명 규칙을 사용하세요. 
모범 사례의 [명명 규칙](../04-agent-skills/03-best-practices.md) 섹션에서 서식 지침을 제공합니다.

각 스킬에 대한 내부 레지스트리를 다음과 함께 유지하세요:

- **목적**: 스킬이 지원하는 워크플로
- **소유자**: 유지 관리를 담당하는 팀 또는 개인
- **버전**: 현재 배포된 버전
- **종속성**: 필요한 MCP 서버, 패키지 또는 외부 서비스
- **평가 상태**: 마지막 평가 날짜 및 결과

### 역할 기반 번들

조직 역할별로 스킬을 그룹화하여 각 사용자의 활성 스킬 세트를 집중시키세요:

- **영업 팀**: CRM 작업, 파이프라인 보고, 제안서 생성
- **엔지니어링**: 코드 검토, 배포 워크플로, 인시던트 대응
- **재무**: 보고서 생성, 데이터 검증, 감사 준비

각 역할 기반 번들은 해당 역할의 일상 워크플로와 관련된 스킬만 포함해야 합니다.

## 배포 및 버전 제어

### 소스 제어

히스토리 추적, 풀 리퀘스트를 통한 코드 검토 및 롤백 기능을 위해 Git에 스킬 디렉터리를 저장하세요. 
각 스킬 디렉터리(SKILL.md 및 번들 파일 포함)는 Git 추적 폴더에 자연스럽게 매핑됩니다.

### API 기반 배포

Skills API는 워크스페이스 범위 배포를 제공합니다. 
API를 통해 업로드된 스킬은 모든 워크스페이스 구성원이 사용할 수 있습니다. 
업로드, 버전 관리 및 관리 엔드포인트는 [API로 스킬 사용](../04-agent-skills/05-using-skills-with-api.md)을 참조하세요.

### 버전 관리 전략

- **프로덕션**: 스킬을 특정 버전으로 고정하세요. 새 버전을 승격하기 전에 전체 평가 스위트를 실행하세요. 모든 업데이트를 전체 보안 검토가 필요한 새로운 배포로 취급하세요.
- **개발 및 테스트**: 프로덕션 승격 전에 변경 사항을 검증하기 위해 최신 버전을 사용하세요.
- **롤백 계획**: 이전 버전을 폴백으로 유지하세요. 프로덕션에서 새 버전이 평가에 실패하면 즉시 마지막으로 알려진 정상 버전으로 되돌리세요.
- **무결성 검증**: 검토된 스킬의 체크섬을 계산하고 배포 시 확인하세요. 스킬 리포지토리에서 서명된 커밋을 사용하여 출처를 보장하세요.

### 크로스 서피스 고려사항

> 커스텀 스킬은 서피스 간에 동기화되지 않습니다. API에 업로드된 스킬은 claude.ai 또는 Claude Code에서 사용할 수 없으며, 그 반대도 마찬가지입니다. 각 서피스는 별도의 업로드 및 관리가
> 필요합니다.

Git에 스킬 소스 파일을 단일 진실 공급원으로 유지하세요. 
조직이 여러 서피스에 걸쳐 스킬을 배포하는 경우, 일관성을 유지하기 위해 자체 동기화 프로세스를 구현하세요. 
자세한 내용은 [크로스 서피스 가용성](../04-agent-skills/01-overview.md)을 참조하세요.
