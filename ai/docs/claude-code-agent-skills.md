## 스킬이란

**스킬(Skill)은 명시적 실행 계약(Execution Contract)입니다.**

단순한 프롬프트 모음이 아니라, 파일 번들과 절차적 지침을 패키징한 모듈형 기능 확장체입니다.

**중요**: 스킬은 실행 계약을 정의하지만, 호출과 지침 해석은 여전히 모델의 의미 기반 판단에 의해 수행됩니다. 스킬은 "강제 실행 엔진"이 아니며, 동일한 입력에 대해 완전히 결정론적인 결과를 보장하지 않습니다.

> **참고: Tool과의 차이**
>
> Claude에는 Tool(함수 호출)이라는 개념도 존재합니다.
> Tool은 특정 함수를 직접 실행하기 위한 인터페이스이며, 단일 작업 수행이나 외부 API 호출에 적합합니다.
>
> 반면 Skill은 파일과 지침을 포함한 모듈형 실행 계약으로, 복잡한 워크플로우나 도메인 지식이 필요한 작업에 사용됩니다.
>
> - 단순 계산, API 호출 → Tool
> - 반복적이고 구조화된 작업 → Skill
>
> 스킬은 파일 형태로 지속되지만, 세션 간 실행 상태를 자동으로 유지하지는 않습니다. 상태가 필요한 경우 외부 저장소를 활용합니다.

## 설계 원칙

스킬 문서는 다음과 같은 설계 원칙을 기반으로 작성합니다.

- **명시성 우선**
  - Claude의 추론에 의존하기보다, 가능한 한 명시적인 절차와 검증 단계를 기술합니다.
- **결정론적 요소를 최대화**
  - 가능한 영역(스크립트 실행 등)에서는 결정론을 확보하고, 모델 추론이 개입되는 영역은 명시적으로 통제합니다.
- **토큰 경제성 고려**
  - 스킬은 컨텍스트 윈도우를 소비합니다.
  - 불필요한 설명, 중복 내용, 대형 참조 파일의 상시 로딩을 지양합니다.
- **점진적 공개**
  - 핵심 지침은 SKILL.md에, 상세 내용은 별도 파일로 분리하여 필요 시 참조합니다.
- **테스트 기반 개선**
  - 스킬은 작성 후 평가 시나리오를 통해 반복 개선합니다.

## 스킬 구조와 파일 배치

### 디렉토리 구조

스킬은 반드시 `.claude/skills/<skill-name>/SKILL.md` 구조를 따릅니다.

```
.claude/
└── skills/
    └── pdf-processing/
        ├── SKILL.md        # 실행 계약 (필수)
        ├── resources/      # 참조 자료 (선택)
        │     └── forms.md
        └── scripts/        # 실행 스크립트 (선택)
              └── analyze.py
```
- SKILL.md
  - 스킬의 실행 계약을 정의하는 핵심 파일입니다.
  - 트리거 조건, 절차, 제약 사항을 명시합니다.
- resources/
  - 참조 문서, 템플릿, 예시 데이터 등 정적 자료를 포함합니다.
- scripts/
  - 필요 시 실행 가능한 스크립트를 포함합니다.
  - 결정론적 처리가 필요한 로직은 이 디렉토리로 분리합니다.

> 참고: `resources/`, `scripts/` 디렉토리는 필수 구성 요소는 아니며, 필요에 따라 선택적으로 구성합니다.
> 
> `.claude/skills/<skill-name>/SKILL.md` 구조는 항상 동일하며, 차이는 배치 위치(적용 범위)에 있습니다. 
> - 사용자 전역 스킬: `~/.claude/` 
> - 프로젝트 전용 스킬: `./project/.claude/` 

### 파일명 규약

**[필수]**

- 소문자 + 하이픈 사용
- 의미 있는 이름 사용
- 공백 사용 금지

```
올바른 예:
- SKILL.md
- finance-schema.md
- analyze-form.py

잘못된 예:
- doc1.md
- My Document.md
- analyzeForm.py
```

## SKILL.md 작성 규약

### 기본 구조

**[필수]**

모든 SKILL.md는 YAML 프론트매터로 시작합니다.

```yaml
---
name: git-commit-message
description: git diff를 분석하여 컨벤션에 맞는 커밋 메시지를 생성합니다. 사용자가 커밋 메시지 작성 또는 변경사항 검토를 요청할 때 사용합니다.
---
```

이후 본문에는 실행 지침을 기술합니다.

### name 필드

**[필수]**

- 소문자, 숫자, 하이픈만 사용
- 64자 이하

**[권장]**

- 동명사 형태 사용 (processing, analyzing)

권장 이유:

- 기능 중심 이름으로 해석이 명확해집니다.
- 자동 트리거 문맥과 자연스럽게 연결됩니다.

### description 필드

**[필수 포함 요소]**

1. 무엇을 수행하는지
2. 언제 사용하는지

잘못된 예:

```
description: 문서를 처리합니다
```

올바른 예:

```
description: PDF에서 텍스트와 표를 추출합니다. 사용자가 PDF 분석 또는 문서 추출을 요청할 때 사용합니다.
```

### 자동 호출(Invocation) 원리

**[필수 이해 개념]**

Claude는 description을 의미 기반 매칭의 주요 신호로 활용하여 사용자 요청과 가장 관련성이 높다고 판단되는 스킬을 선택합니다.

**작동 메커니즘**

1. 사용자가 요청을 입력하면, Claude는 등록된 모든 스킬의 description을 의미 기반으로 분석합니다.
2. description의 "무엇을"과 "언제"가 사용자 요청 의도와 의미적으로 매칭되는 스킬을 선택합니다.
3. 매칭된 스킬의 SKILL.md를 로딩하여 지침을 실행합니다.

> 참고: 단순한 키워드 검색이 아닌 의미 기반 매칭(semantic matching)을 사용합니다.

**description이 중요한 이유**

| 문제 유형 | 원인 | 결과 |
|---------|------|------|
| **오탐** | description이 너무 모호함 | 관련 없는 스킬이 호출됨 |
| **미탐** | description이 너무 좁음 | 필요할 때 호출되지 않음 |
| **충돌** | 여러 스킬이 유사한 description | 예측 불가능한 선택 |

> **참고**: 현재 스킬 간 명시적 우선순위를 직접 지정하는 기능은 없습니다. description 설계가 현재 가장 중요한 제어 수단입니다. 단, 이는 확률적 매칭이므로 완전한 제어를 보장하지 않습니다.

**description 작성 전략**

- **구체적이면서도 넓게**: 특정 기능은 명확히, 사용 시점은 다양한 표현을 고려
- **중복 최소화**: 조직 내 다른 스킬과 중복되지 않도록 유니크한 키워드 사용
- **키워드 최적화**: 사용자가 실제로 사용할 검색어를 포함

**예시 비교**

```yaml
# ❌ 오탐 위험: 너무 모호
description: 코드를 분석합니다

# ❌ 미탐 위험: 너무 좁음
description: git diff --staged를 실행하여 커밋 메시지를 생성합니다

# ✅ 적절한 범위
description: git diff를 분석하여 컨벤션에 맞는 커밋 메시지를 생성합니다. 사용자가 커밋 메시지 작성 또는 변경사항 검토를 요청할 때 사용합니다.
```

### 스킬 충돌 관리 전략

**[필수 이해 개념]**

조직 내 여러 스킬이 유사한 description을 가질 경우, 자동 호출에서 충돌이 발생할 수 있습니다.

**충돌 증상**

- 동일 요청에 대해 번갈아 다른 스킬이 호출됨
- 예상한 스킬이 호출되지 않음
- 사용자 요청 의도와 다른 스킬이 선택됨

**충돌 예방 전략**

| 전략 | 설명 | 적용 시점 |
|-----|------|----------|
| **네이밍 컨벤션** | 조직/도메인 접두사 사용 (finance-report, hr-onboarding) | 스킬 생성 시 |
| **키워드 분리** | 유사한 스킬 간 description 키워드 중복 최소화 | description 작성 시 |
| **범위 명확화** | 각 스킬의 고유 사용 시점을 명확히 구분 | 설계 시 |
| **통합 고려** | 유사 기능은 하나의 스킬로 병합 검토 | 스킬 다수 존재 시 |

**충돌 해결 절차**

1. **진단**: 어떤 요청에서 어떤 스킬이 경쟁하는지 기록
2. **분석**: description의 의미적 중첩 영역 식별
3. **조정**:
   - 한쪽 스킬의 description을 더 구체화하거나
   - 두 스킬을 통합하거나
   - 사용 시점을 명확히 분리
4. **검증**: 테스트 시나리오로 충돌 해결 확인

### 추론 통제 전략

**[권장]**

Claude는 과도하게 추론하려는 경향이 있습니다. 스킬은 이를 제어하기 위한 실행 계약입니다.

**추론 허용 범위 정의**

| 자유도 | 허용 범위 | 지시문 강도 |
|-------|----------|-----------|
| **없음** | 정확히 지정된 명령만 | "정확히 이 명령을 실행하세요" |
| **낮음** | 파라미터 선택 가능 | "--verify 플래그는 항상 포함" |
| **중간** | 템플릿 내 커스터마이징 | "템플릿을 기준으로 상황에 맞게 조정" |
| **높음** | 원칙 기반 판단 | "코드 품질 원칙에 따라 검토" |

**지시문 강도 조절 방법**

```markdown
# ❌ 애매한 지시

파일을 처리하세요.

# ✅ 명확한 지시

1. scripts/validate.py 실행
2. 오류 없으면 scripts/process.py 실행
3. 오류 있으면 중단하고 보고
```

## 내용 분리 전략

### 파일 크기 기준

**[권장] 500줄 기준**

SKILL.md는 500줄 이하로 유지합니다.

> 참고: 500줄은 일반적으로 수천 토큰 내로 유지하기 위한 실무적 기준이며, 팀 합의 기반 운영 기준입니다.

이 기준의 목적:

- 컨텍스트 과다 점유 방지
- 유지보수성 향상
- 인지 부하 감소

**[필수] 800줄 운영 상한**

800줄을 초과하는 경우 파일 분리는 권장이 아닌 필수입니다. 이는 컨텍스트 비용 및 유지보수 위험이 높아지는 임계점으로, 운영 안정성을 위한 강제 정책입니다.

| 기준 | 라인 수 | 수준 | 조치 |
|-----|--------|------|------|
| 권장 | 500줄 이하 | 권장 | 유지 |
| 경고 | 500~800줄 | 검토 필요 | 분리 검토 |
| 필수 | 800줄 초과 | 필수 | 즉시 분리 (운영 상한) |

### 점진적 공개와 파일 로딩 메커니즘

> **중요**: 본 문서의 로딩 모델은 실제 Claude 내부 구현을 설명하는 것이 아니라, **운영 상 예측 가능성을 확보하기 위한 개념적 추상화**입니다. 실제 플랫폼 동작은 구현에 따라 다를 수 있으며, 이 모델은 팀 내 일관된 이해와 운영 기준 수립을 위한 설계 도구입니다.

개념적으로 스킬은 **3단계 로딩 패턴**을 가집니다.

| 레벨 | 로딩 시점 | 컨텍스트 사용 | 포함 콘텐츠 |
|-----|----------|-------------|-----------|
| **레벨 1** | 항상 (시작 시) | 소량의 메타데이터 | name, description (메타데이터) |
| **레벨 2** | 스킬 호출 시 | SKILL.md 전체 | SKILL.md 본문 (지침) |
| **레벨 3+** | 필요 시 (참조 시) | 호출 시 로드 | 번들 파일 (스크립트, 참조 자료) |

> **참고**: 정확한 토큰 비용은 모델 및 플랫폼 버전에 따라 달라질 수 있습니다.

**로딩 동작 원리**

1. **시작 시**: 모든 스킬의 메타데이터(name, description)만 로드
2. **호출 시**: description 매칭된 스킬의 SKILL.md 본문 로드
3. **참조 시**: SKILL.md 내 링크된 파일만 추가 로드

**참조 깊이 규칙**

> 참고: 마크다운 링크와 자동 로딩은 다릅니다. SKILL.md에서 명시적으로 참조한 파일은 호출 맥락에 따라 추가 로딩될 수 있습니다.

```
원칙: 1단계 참조까지만 허용
SKILL.md → FORMS.md (✅ 허용)
SKILL.md → REFERENCE.md (✅ 허용)

예외: 2단계 이상 참조는 금지
SKILL.md → FORMS.md → DETAILS.md (❌ 금지)
```

> **중요**: 이 제한은 플랫폼 제약이 아니라 운영 안정성을 위한 팀 정책입니다. 실제 로딩 전략은 모델 및 플랫폼 구현에 따라 달라질 수 있습니다.

**1단계 제한의 이유**

- **예측 가능성**: 로딩되는 콘텐츠를 명확히 제어
- **컨텍스트 관리**: 예상치 못한 대량 로딩 방지
- **디버깅 용이성**: 어떤 파일이 로드되는지 추적 가능
- **성능**: 중첩 탐색으로 인한 지연 방지

**[예외 처리]**

복잡한 도메인에서 2단계 참조가 필요한 경우:

### 외부 상태 관리 전략

**[필수 이해 개념]**

**상태 관리 패턴**

| 패턴 | 사용 사례 | 구현 방법 |
|-----|----------|-----------|
| **파일 기반** | 누적 결과, 설정 | JSON/YAML 파일 읽기/쓰기 |

**주의사항**

**운영 수준 전략 예시: 원자적 쓰기**

파일 기반 상태에서 부분 실패 방지:

```python
import os
import json

def atomic_write(filepath, data):
    temp_path = filepath + ".tmp"
    with open(temp_path, 'w') as f:
        json.dump(data, f)
        f.flush()
        os.fsync(f.fileno())  # 디스크 동기화
    os.replace(temp_path, filepath)  # 원자적 교체
```

- 쓰기 중단 시 임시 파일만 남음
- 손상된 상태 파일 방지
- 롤백 시 `.bak` 복사본 활용

- 모든 예외는 문서화 및 감사 로그 기록

#### 공통 모듈 관리

중복을 피하면서도 1단계 참조 규칙을 준수하는 전략:

**방법 1: 템플릿 스킬 (권장)**

```
.claude/skills/_templates/
├── error-handling.md      # 공통 오류 처리 패턴
├── validation-patterns.md  # 검증 템플릿
└── logging-standards.md   # 로깅 표준
```

각 스킬은 템플릿을 복사하여 사용하되, 직접 참조하지 않습니다. 복사 시점의 버전을 주석에 기록합니다.

> **설계 철학**: 예측 가능성과 참조 깊이 통제를 위해 DRY(Don't Repeat Yourself) 원칙보다 명시성을 우선합니다. 중복된 코드가 있더라도 각 스킬의 독립성과 로딩 예측 가능성이 더 중요합니다.

**방법 2: 스킬 간 계층 구조 (개념적)**

기본 스킬(Base Skill)과 특화 스킬(Specialized Skill)의 개념적 구분:

| 계층 | 역할 | 예시 |
|-----|------|------|
| L1: 기반 | 공통 유틸리티 | `data-processing` |
| L2: 도메인 | 특정 도메인 처리 | `finance-report`, `hr-onboarding` |
| L3: 작업 | 단일 작업 수행 | `generate-monthly-report` |

#### 스킬 수량 거버넌스

다음 기준은 예시입니다. 조직의 스킬 생태계 복잡도에 맞게 조정하세요.

**통합 기준**

다음 경우 스킬을 통합하는 것을 검토합니다:

- 유사한 description을 가진 스킬들이 경쟁적으로 호출됨
- 스킬들이 반복적으로 함께 호출되는 패턴이 관찰됨
- 다수의 스킬이 동일한 참조 파일을 공유함

**분해 기준**

다음 경우 스킬을 분해하는 것을 검토합니다:

- SKILL.md가 800줄 상한을 초과 (필수 분해)
- 명확히 구분되는 독립적인 사용 시점이 다수 존재
- 특정 사용 시점에서만 필요한 참조 파일이 과다함

### 완료 체크리스트

#### 메타데이터

- name 규약 준수
- description에 기능 + 사용 시점 포함
- 자동 호출 테스트 (description이 적절히 매칭되는지)

#### 구조

- 500줄 이하
- 중첩 참조 없음 (1단계까지만)
- 점진적 공개 적용

#### 코드

- 예외 명시 처리
- 민감정보 없음
- 상수 정당화

#### 테스트

- Haiku 테스트 (최소 리소스)
- Sonnet 테스트 (표준)
- Opus 테스트 (복잡한 시나리오)
- 자동 호출 테스트
  - Positive: 의도한 요청에서 해당 스킬이 호출되는지
  - Negative: 다른 의도에서 해당 스킬이 호출되지 않는지
  - Ambiguous: 경계 문장에서 의도한 대로 동작하는지

### 스킬이 적합하지 않은 경우

**[참고]**

모든 작업을 스킬로 만들 필요는 없습니다. 다음 경우는 스킬화를 재고려하세요:

| 경우 | 이유 | 대안 |
|-----|------|------|
| **일회성 작업** | 재사용성이 없음 | 단순 프롬프트 |
| **단순 계산** | 스킬 오버헤드보다 비효율 | 직접 계산 또는 도구 사용 |
| **단일 함수 호출** | 스킬의 복잡성이 불필요 | 직접 함수 호출 |
| **빈번히 변경** | 유지보수 부담 증가 | 동적 프롬프트 또는 설정 파일 |

### 버전 표기

**[권장]**

```yaml
---
name: pdf-processing

---
```

### Breaking Change 관리

**[필수]**

하위 호환성이 깨지는 변경 시:

1. 버전을 주(major) 버전 업그레이드
2. 변경사항을 CHANGELOG.md에 기록
3. 기존 동작을 유지하는 마이그레이션 가이드 제공
