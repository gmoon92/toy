# 상세 비교 — DPoP(클라 증명) vs 티켓(서버 nonce) : "왜 nonce 방식인가"

> 대상: `pns-auth-complete.html`(DPoP 착안) vs `~/Downloads/session-ticket-binding.md`(cnf+PoP)
> 핵심 질문: 두 방식은 `cnf` 바인딩까지 동일한데, 왜 티켓은 "중계가 nonce를 보내 서명받는" 방식으로 가는가?
> 작성: 2026-07-03

---

## 0. 결론 (TL;DR)

- 두 방식의 차이는 **딱 하나 — "증명의 신선도(freshness)를 누가 보증하느냐"** 뿐이다.
  - DPoP = 클라가 `iat`(시각)+`jti`(난수)로 **주장**, 서버는 시간창+캐시로 신뢰.
  - 티켓 = 서버가 **nonce를 발급**해 신선도를 **검증**(challenge-response).
- **접근 대상이 API/URL이 아니라 "연결(핸드셰이크)"이면 DPoP의 핵심(`htm/htu` 요청 바인딩)이 빠지고, 증명이 밋밋한 "시각 찍힌 서명"으로 전락**한다. 남는 리플레이 방어가 `iat` 시간창 + `jti` 캐시뿐인데 둘 다 약하다.
- 연결은 **어차피 왕복(CONNECT↔CONNACK)을 한다.** DPoP의 iat/jti는 "왕복 회피용" 우회책인데, 회피할 왕복이 이미 있으므로 그 비용(시계 의존·분산 캐시·리플레이 창)을 치를 이유가 없다.
- ∴ 이미 하는 왕복에 nonce를 얹는 challenge-response가 **엄밀히 우월**하고, 그래서 티켓 방식으로 간다.

---

## 1. 두 방식이 "같은" 부분 — cnf 바인딩(=이중 서명)

문겸 님이 착안한 구조와 티켓 설계는 아래가 **동일**하다. 여기엔 이견이 없다.

```
① 클라: 임시 키쌍 생성. 개인키는 기기 밖으로 안 나감(비추출). 공개키만 서버에 제출
② 서버(BE/iam): 공개키 지문(jkt=SHA-256 thumbprint)을 토큰 cnf에 박아 서명   ← 1차 서명
③ 클라: 접속 시 개인키로 증명(proof) 서명                                    ← 2차 서명
④ 검증자(중계): 토큰 서명 검증 + proof 서명 검증 + SHA256(proof 공개키)==토큰.cnf.jkt 대조
```

- 이 sender-constrained(발신자 구속) 성질 덕분에 **토큰만 훔쳐도 개인키가 없어 못 쓴다.**
- 표준 근거: `cnf` 클레임 = RFC 7800(PoP key semantics). DPoP(RFC 9449)·mTLS-bound(RFC 8705) 모두 이 위에 얹은 것.
- **즉 "이중 서명(cnf 박제 + 클라 증명)"은 두 방식의 공통 토대다. 논쟁 대상이 아니다.**

## 2. 두 방식이 "다른" 유일한 부분 — 신선도 보증 메커니즘

| | DPoP 방식 (클라 증명) | 티켓 방식 (서버 nonce) |
|:--|:--|:--|
| 증명 페이로드 | `{htm, htu, iat, jti}` 서명 | `{server_nonce, jti}` 서명 |
| 신선도 근거 | 클라가 **주장**한 `iat` + 서버 `jti` 캐시 | 서버가 발급한 nonce (서버가 **보증**) |
| 서버 왕복 | 불필요(원래 목표) | 1왕복 (핸드셰이크에 포함) |
| 시계 동기화 | 필요(`iat` 창 판정, 클라 시계 신뢰) | 불필요 (서버가 시각을 봉인하거나 저장) |
| 서버 상태 | `jti` 집합, **전 인스턴스 공유(Redis) 강제**, 요청량 비례 | nonce, **발급 인스턴스 로컬**, TTL 바운드 (또는 상태 없이 HMAC) |
| 리플레이 창 | `iat` 창(예 ±60초) 동안 존재 | nonce 1회성 소비/연결 바인딩으로 닫힘 |

> **주의(정정):** 초안에서 nonce를 "상태 0 / 캐시 0"으로 표현한 것은 과장이었다. **nonce도 서버가 상태(또는 최소 비밀키)를 갖는다.** 진짜 차이는 "상태 유무"가 아니라 **상태의 종류** — 아래 §2.1 참조.

### 2.1 "상태 유무"가 아니라 "상태의 종류"다 (핵심 정정)

두 방식 다 서버가 상태를 갖는다. 차이는 **누가 값을 만들고, 그 상태를 노드 간 공유해야 하느냐**다.

| | DPoP `jti` 캐시 | nonce 상태 |
|:--|:--|:--|
| 값 생성 주체 | **클라**가 골라 보냄(unsolicited) | **서버**가 발급 |
| 공유 범위 | **전 인스턴스 공유 필수(Redis)** | **발급 인스턴스 로컬** |
| 크기 | 요청량 비례(본 jti 다 기록) | 미사용 challenge만(짧은 TTL) |

- **DPoP가 공유를 강제하는 이유**: 클라가 고른 jti를 가진 리플레이가 **아무 노드로나** 갈 수 있다(HTTP는 요청마다 LB). 노드 B는 노드 A가 그 jti를 봤는지 모르니, 막으려면 전 노드가 jti 집합을 공유해야 한다 → Redis.
- **nonce가 로컬로 충분한 이유**: 핸드셰이크(nonce 발급→서명→검증)가 **한 TCP 연결·한 인스턴스 안에서** 끝난다. 노드 A가 발급한 nonce는 노드 A가 회수. 공격자가 가로챈 증명으로 새 연결을 열면 노드 B로 갈 수 있고, **노드 B는 그 nonce를 발급한 적 없어 거부**한다. → 노드 간 공유 불필요.

### 2.2 "방금 만들었다"를 어떻게 보장하나 — 구현 두 가지

**(A) 상태 있는 nonce (서버 저장):**
```
nonce = 256비트 난수. 서버가 저장(TTL 예: 30초) 후 발급.
회수: 내 저장소에 있나 + 미소비인가? → 소비(삭제) → 통과.
```
- 보장: 그 난수가 **내 저장소에 있다 ⟺ 내가 최근(TTL 내) 넣었다.** 위조 확률 1/2²⁵⁶. 한 번 쓰면 삭제 → 재전송 즉사.

**(B) 상태 없는 nonce (서버 서명, 저장 안 함):**
```
nonce = timestamp ‖ HMAC(서버비밀키, timestamp ‖ 연결식별자)
회수: HMAC 재계산 일치? → 내가 발급한 게 맞다(공격자는 키 없어 위조 불가)
     + 박힌 timestamp가 창 안인가? → 신선하다
```
- 보장: **timestamp를 서버가 HMAC으로 봉인** → 그 시각은 클라 주장이 아니라 **서버 보증**. 저장소 불필요, 서버 비밀키 1개만. (SYN 쿠키·TLS stateless ticket 원리)

**DPoP `iat`와의 결정적 차이:** DPoP는 시각을 **클라가 주장** → 서버가 검증 불가 → 창+jti캐시로 보완(클라 시계 신뢰). nonce는 (A)든 (B)든 신선도 근거가 **전부 서버 쪽**(자기 저장소 또는 자기 비밀키) → **클라 시계를 한 번도 믿지 않는다.**

### 2.3 "중계는 DB가 없다"(§6) 제약과의 관계 — 오히려 nonce에 유리

"DB 없음" = 조회용 **공유·영속 저장소**가 없다는 뜻이지, **메모리 상태가 없다**는 게 아니다. 중계는 이미 열린 소켓·MQTT 세션·keepalive 타이머·구독 목록을 인메모리로 들고 있다. nonce는 그 "이미 있는 상태"에 얹힌다.

- **(B) stateless HMAC nonce → 저장소 0.** 서버 비밀키 1개만(중계가 이미 캐시하는 JWKS와 같은 설정값 범주). 회수 시 HMAC 재계산 = DB 조회가 아니라 CPU 연산. **"DB 없는 중계"의 정답.**
- **(A) 인메모리 nonce → DB 아니라 소켓 컨텍스트.** nonce를 소켓에 내리고 몇 초 뒤 같은 소켓으로 응답이 오므로, 그 연결 객체 메모리에 잠깐 물고 있으면 끝. 핸드셰이크 종료 시 소멸.

**오히려 "DB 없음"이 DPoP에 불리하다:**
- DPoP `jti` 캐시는 클러스터 전체 중복 제거를 위해 **공유 Redis를 원함 → "DB 없음" 위반**(SPOF 재도입).
- nonce(B)는 저장소 0으로 "DB 없음"에 완벽 부합. → §6 제약은 **stateless nonce와 맞고 DPoP jti 캐시와 충돌**한다.

**정직한 단서 — 딱 하나는 공유 스토어를 원한다:**
- **단일사용·즉시폐기(redeem, T3/T4)** 는 신선도(nonce)와 별개 층이다. "클러스터 전체 1회 사용" 또는 "발급 후 즉시 무효화"를 원하면 공유 원자 소비 스토어(Redis)가 진짜 필요하다.
- 그러나 이건 **DPoP를 써도 "즉시 폐기"를 원하면 똑같이 필요** → "DB 없음"이 어느 쪽도 편들지 않는 공통 비용.
- 중계가 공유 스토어를 절대 못 두면: redeem(단일사용) 포기 + **PoP 신선도만 stateless nonce(B)** → T5(가로채기 재전송)까지 DB 없이 방어, T3(2회 사용)만 감수하는 선택.

> 요약: **PoP 신선도(nonce)는 DB 없이 된다(B는 저장소 0). DB가 필요한 건 "단일사용/즉시폐기"라는 별도 욕심이고, 그건 DPoP도 마찬가지다.**

## 3. 왜 DPoP가 이 케이스에서 반쪽이 되는가 (URL이 없다)

DPoP의 진짜 방어력은 `htm`(메서드)+`htu`(URL)로 증명을 **"이 요청 전용"으로 묶는** 데서 나온다.
- HTTP 리소스 서버: `POST /api/transfer`용 증명은 `GET /api/balance`에 못 쓴다 → 리플레이 표면이 URL 단위로 쪼개짐.
- **연결(핸드셰이크): 붙는 엔드포인트가 사실상 하나**라 htm/htu가 좁혀주는 게 없다. 넣을 URL 자체가 마땅치 않다(문겸 님 지적).

htm/htu가 무의미해지면 DPoP 증명은 아래로 **전락**한다:

```
{ typ:"dpop+jwt", jwk:공개키 } . { iat: <시각>, jti: <난수> } . <서명>
                                   └─ 남은 건 "시각 찍힌 서명 한 장" ─┘
```

이제 리플레이를 막을 장치가 **`iat` 시간창 + `jti` 캐시 둘뿐**이다.

## 4. 그 남은 두 장치가 왜 약한가 — 공격 워크스루

### 시나리오 T5: 가로채기 후 만료 전 재전송 (원격제어에서 특히 위험)

전제: 개인키는 못 훔침(비추출). 하지만 클라가 만든 **증명 자체를 전송 중 가로챔**
(MITM — 특히 현재 코드 SSL이 trust-all이라 현실적. `MqttIntegrationConfig.java:144-155`).

**DPoP(클라 증명)일 때:**
```
1. 클라가 proof_1 = sign({iat=T, jti=J1}) 생성해 전송
2. 공격자가 proof_1을 가로챔 (복제)
3. 공격자가 T+30초 시점에 proof_1을 그대로 재전송
   → iat 시간창(±60초) 안 → 통과
   → jti 캐시? 다중 인스턴스면 다른 노드로 보내 우회 → 통과
4. 공격자, 정당한 세션 탈취
```
- 방어하려면: 시계 동기화 + **모든 중계 노드가 공유하는 jti 스토어(Redis) + 원자적 조회·기록**.
  이건 §6이 없애려던 "공유 상태·서버 왕복"을 되살린다. 게다가 여전히 창(60초) 리플레이 여지는 남음.

**티켓(서버 nonce)일 때:**
```
1. 클라가 중계에 접속 요청
2. 중계가 nonce_X 발급 (방금 만든 1회성 난수, 서버가 기억)
3. 클라가 proof = sign({nonce_X, jti}) 전송
4. 공격자가 proof를 가로채도 → nonce_X는 이 연결에 소비됨/만료
   → 재전송 시 서버가 "nonce_X는 이미 썼다/모른다"로 즉시 거부
   → 리플레이 구조적으로 불가
```
- 시계 동기화 불필요(서버가 nonce 수명 관리). 분산 jti 캐시 불필요(nonce 소비가 대체).

> 결론: htm/htu가 빠진 상태에서 iat/jti는 T5를 못 막는다. nonce는 T5를 **구조적으로** 막는다.

## 5. 그런데 왜 DPoP는 처음부터 nonce를 안 쓰나? (오해 해소)

DPoP가 iat/jti를 택한 건 **약해서가 아니라 목표가 달라서**다.

- DPoP의 설계 목표 = **"stateless HTTP에서 요청 수천 건을 서버 왕복 없이 인증"**.
  - 요청마다 서버가 nonce를 내려주려면 요청마다 왕복이 생겨 성능이 죽는다.
  - 그래서 "클라가 시각을 주장 + 서버가 창+캐시로 걸러냄"이라는 **왕복 회피용 우회책**을 쓴다.
- 그 대가: 시계 의존 + 리플레이 캐시 + 잔여 리플레이 창.
- RFC 9449도 이 약점을 인지해 **`DPoP-Nonce`(서버 nonce) 옵션을 표준에 추가**했다. 즉 "더 강하게 하려면 결국 nonce로 가라"는 게 표준의 입장.

**핵심 통찰:** 연결 핸드셰이크는 **애초에 왕복(CONNECT↔CONNACK)을 한다.**
DPoP가 피하려던 그 왕복이 여기선 이미 일어난다. → 회피할 이유가 사라진 우회책(iat/jti)을 굳이 쓰는 건 손해. 이미 하는 왕복에 nonce 한 줄 얹는 게 순이득.

## 6. "정석"의 정확한 경계

| 상황 | 정석 | 이유 |
|:--|:--|:--|
| HTTP 리소스 서버, 다중 요청, 왕복 회피 필요 | **DPoP** (htm/htu + iat/jti) | 요청 단위 바인딩 + 왕복 0 |
| 연결 핸드셰이크(1회 인증), 왕복 이미 존재 | **서버 nonce challenge-response** | 신선도 자명·시계/캐시 불필요 |

- TLS·SSH·Kerberos·FIDO/WebAuthn = 전부 후자(challenge-response). "연결을 세운다"는 문제의 표준형.
- 문겸 님 상황(접근 대상=연결, URL 아님) → **후자가 정석.** 티켓 문서가 그리로 간 이유가 이것.

## 7. 그래도 남는 공통 과제 (어느 방식이든)

- **장수명 연결 사각지대**: nonce든 iat/jti든 **핸드셰이크 1회만** 증명한다. 이후 유지 세션은 TLS 무결성에 의존. → 세션 하이재킹 방어는 별도 층(강제 종료 채널·세션 탈취 탐지).
- **MQTT v3.1.1(Paho mqttv3) 적재 문제**: 소스상 헤더/User Property 없음 → nonce 왕복이든 이중 헤더든 **프로토콜에 실을 여지가 필요**(MQTT5 여부·CONNECT 필드 확장). 티켓 문서는 이걸 중계팀 Q1으로 정직히 질문함.
- **TLS 인증서 검증**: 현 코드 trust-all → challenge-response의 전제(전송 무결성)도 흔들린다. P0에서 인증서 검증 강제 필수.

## 8. 한 줄 요약

> `cnf`(이중 서명)는 두 방식이 같다. 다른 건 신선도 보증뿐이다.
> URL이 없어 htm/htu가 빠지면 DPoP는 "시각 찍힌 서명"으로 약해지고, 그 신선도는 iat창+jti캐시(=시계 의존+분산 SPOF+리플레이 창)에 의존한다.
> **연결은 어차피 왕복하므로, 그 왕복에 서버 nonce를 얹는 challenge-response가 시계·캐시·창을 전부 0으로 만든다 — 그래서 티켓(nonce) 방식으로 가는 것이 맞다.**
