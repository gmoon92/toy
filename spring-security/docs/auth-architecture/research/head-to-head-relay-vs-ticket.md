# Head-to-Head — 릴레이 TCP(cnf+ath+redeem) vs 접속 티켓(cnf+서버 nonce)

> 두 소유증명(PoP) 설계의 정밀 비교·통합 권고. 실제 문서 본문 인용에 근거함.
> 대상 A: `dpop-on-tcp-relay-architecture.md` · 대상 B: `session-ticket-binding-design.md`
> 작성: 2026-07-06 (3-에이전트 병렬 정밀 분석 종합)

---

## 0. 결론 (TL;DR)

- cnf **이중 서명**과 **redeem(단일사용)**은 두 문서 **공통**이다. 논쟁 대상이 아니다.
- 진짜 차이는 오직 하나 — **"검증 서버가 T5(선점 재전송)를 어떻게 막느냐"**:
  - **A** = T5를 **TLS(전송 계층)에 위임**하고 앱은 **무왕복(1-RTT)** 유지. 서버 nonce는 "TLS 붕괴 대비 선택적 심층방어".
  - **B** = T5를 **서버 nonce(앱 계층)로 자체 방어**. nonce가 기본 핵심층(**2-RTT**).
- **TLS+인증서 검증이 실제로 켜져 있으면 A(무왕복)가, 현재처럼 trust-all이면 B(nonce)가 우세.**
- role·aud·force_disconnect·전송 근본원인 등은 **택일이 아니라 범위 차이 → 합치면 사라진다.**

---

## 1. 공통 토대 (논쟁 대상 아님)

| 공통 요소 | A | B |
|---|---|---|
| 클라 임시 키쌍(개인키 비추출) | "개인키는 기기 메모리 밖으로 절대 나가지 않는다" | "개인키는 비추출 저장(HW keystore / WebCrypto non-extractable)" |
| `cnf.jkt` = 공개키 SHA-256 지문 박제 | "cnf.jkt 박제해 토큰 발급" | "`cnf` = 제시자 공개키 지문(`jkt`)" |
| 접속 시 개인키 Proof → 검증자 크로스 매칭 | "SHA256(pop_proof.jwk) == access_token.cnf.jkt" | "PoP 공개키 지문 == 티켓 cnf" |
| **redeem(단일사용) 별도 층 필수** | "redeem만 공유 스토어를 쓴다" | "redeem은 바인딩을 얹어도 여전히 필요" |

발급자 서명 알고리즘만 미세 차이: **A = ES256(EC P-256)**, **B = 티켓은 iam의 EdDSA + PoP만 ES256**.

---

## 2. 핵심 차이 — 신선도(T5)를 누가 막는가

> ⚠️ **자주 틀리는 요약 "A=클라 iat, B=서버 nonce"는 현재 A에 대해 틀림.** A는 iat 창 의존을 명시적으로 부정한다: *"Proof 자체의 iat 창·jti 캐시에 의존하지 않는다 … 리플레이 방어의 축이 '토큰 단일사용'으로 옮겨간다."*

| | A (릴레이 TCP) | B (접속 티켓) |
|---|---|---|
| Proof 페이로드 | `{ ath=SHA256(토큰), iat }` | `Sign(nonce ‖ jti)` |
| 신선도 근거 | **토큰 exp + redeem 단일사용** (iat 창 아님) | **서버 발급 nonce** (1회성) |
| 신선도 보증 주체 | 발급자(exp) + 검증자(redeem) | 검증자(중계, nonce) |
| 서버 왕복 | **0왕복** (토큰+Proof 1회 동시 전송) | **1왕복** (nonce challenge) |
| 클라 시계 의존 | 없음(iat는 방어 축 아님) | 없음(서버가 nonce 관리) |

**결정적 판정:** 차이는 "iat vs nonce"가 아니라 **"T5(공격자가 정당 클라보다 먼저 1회 제출)를 누가 막느냐"**다.
- A: T5를 **TLS에 위임**, 앱은 왕복 0 유지. nonce는 선택적 심층방어(A도 "이 서버 nonce가 유일한 진짜 서버 챌린지"라고 인정).
- B: T5를 **nonce(앱 계층)로 자체 방어**.

즉 두 문서는 **동일한 nonce 메커니즘의 위상만 다르게 배치**(A=옵션, B=기본)한 것이다.

## 3. 상태·DB·왕복(RTT)

| 항목 | A | B |
|---|---|---|
| 서명/cnf/ath 검증 | 순수 CPU 무상태 | 무상태(JWKS 오프라인) |
| 공유 스토어 필요 시점 | **redeem만** | **redeem** (+ nonce는 로컬/HMAC) |
| 왕복 수 | **1-RTT** | **2-RTT** |

**공유 스토어 부담은 실질 무승부** — 양쪽 다 redeem 때문에 필요하다. (`detailed-comparison.md §2.3`: "DB가 필요한 건 단일사용/즉시폐기라는 별도 욕심이고, 그건 어느 쪽도 마찬가지.") 차이는 **왕복 수(A 1 < B 2)**.

## 4. 위협 커버리지 (T1~T5)

| 위협 | A | B | 근거 |
|---|:--:|:--:|---|
| T1 위조 티켓 | ✅ | ✅ | 서명 검증 |
| T2 만료 후 재생 | ✅ | ✅ | 짧은 exp |
| T3 2회+ 재사용 | ✅ | ✅ | redeem |
| T4 발급 후 즉시 폐기 | ✅ | ✅ | redeem 원자 소비 |
| **T5 선점 재전송 · TLS 정상** | ✅(TLS가 막음) | ✅(nonce) | 무승부, RTT는 A 우위 |
| **T5 선점 재전송 · TLS trust-all** | ❌(기본)/선택 nonce 시 ✅ | ✅ | **B 우위** |

> 🔴 **현실 점검:** `research/dpop-security-review.md`에 따르면 현재 MQTT SSL이 **trust-all**(`checkServerTrusted()` 비어 있음). 즉 A가 딛고 선 "TLS+인증서 검증" 전제가 **현재 거짓**이므로, **현 코드베이스 기준 T5는 B(nonce)가 우세.** 단 이는 A 설계 결함이 아니라 **미충족 전제(P0)** — TLS 실제 적용·인증서 검증·클라 ssl 사용을 반드시 교차 검증해야 한다.

## 5. 바인딩 외 차이 — 택일이 아니라 범위 차이 (합치면 사라짐)

| 축 | A | B |
|---|---|---|
| 전송 근본원인·TLS 직교성 | **전면 전개** | 거의 없음 |
| room id 사고 근본원인 규명 | **명시** | 없음 |
| MQTT 토픽 구독 authz | **있음** | 없음 |
| role 방향성(viewer↔agent 격리) | 없음 | **있음** |
| aud(AT 혼용 차단) | 언급만 | **있음(선행조건)** |
| force_disconnect(강제 종료) | 없음 | **있음** |
| 동시접속 한도 | 없음 | **있음** |
| 구현 실현성 질문(중계팀) | 없음 | **있음(Q1~Q5)** |

**A = 전송축 + 사고 대응 + 토큰 하드닝 로드맵**, **B = 바인딩층 정밀 계약 + 운영 거버넌스 + 실현성 질문.**

## 6. 흔한 오해·오류 경고

1. **"A=클라 iat, B=서버 nonce"는 틀림** — A의 신선도 축은 exp+redeem. iat는 방어 축이 아님.
2. **`research/` 3문서를 현재 A에 그대로 대입 금지** — 그 문서들의 대상은 A의 구버전(`pns-auth-complete.html`). 현재 A는 htm/htu 제거·jti캐시→redeem 전환·감염단말 범위밖 명시로 이미 개정됨.
3. **"B는 nonce라 DB 불필요"는 틀림** — 양쪽 다 redeem 때문에 공유 스토어 필요.
4. **"redeem 있으면 nonce 불필요"는 틀림** — redeem은 T3(2회 소비)를 막지만 **T5(선점 1회)는 못 막음**(첫 소비자가 공격자가 됨). T5는 nonce 또는 TLS(또는 채널 바인딩)만 막음.
5. **"A는 DPoP라 더 표준"은 틀림** — A는 DPoP의 cnf 아이디어만 차용, htm/htu·DPoP-Nonce는 버림. 순수 DPoP 아님.
6. **두 문서는 같은 채널을 다루지 않음** — A는 데이터 중계(순수 TCP 프레임), B는 시그널링(레거시 MQTT 가능성 포함). "MQTT에 실을 자리 없음" 문제는 B/시그널링엔 유효, A/데이터중계엔 프레임 자유도가 오히려 높음.

## 7. 항목별 우열 (전제 포함)

| 항목 | 우열 |
|---|---|
| cnf 이중 서명 / redeem | 무승부 (공통) |
| T5 · TLS+인증서 정상 | 무승부 (A는 1-RTT 이점) |
| **T5 · TLS trust-all (현 코드)** | **B 우위** |
| 왕복(RTT) | **A 우위** (1 vs 2) |
| 공유 스토어 부담 | 실질 무승부 |
| 전송축·근본원인·MQTT 토픽 authz | **A 우위** |
| role·aud·force_disconnect·동시접속·실현성 | **B 우위** |

## 8. 통합 권고 — 하나로 합친다면

1. **전송(P0)**: A 축 — 데이터 중계 `tcp://`→`ssl://` + **서버 인증서 검증 강제(trust-all 제거)** + **클라가 실제 ssl 사용하는지 교차 검증**.
2. **바인딩(cnf)**: 공통 — `cnf.jkt` 박제 + 접속 시 개인키 PoP.
3. **단일사용(redeem)**: 공통 — 공유 스토어 `jti` 원자 compare-and-delete.
4. **신선도(T5)**: 전제별 분기 —
   - TLS+인증서 확실한 데이터 중계(TCP) → **A식 무왕복(exp+redeem)**.
   - TLS 신뢰 약하거나 고위험·레거시 시그널링 → **B식 서버 nonce**.
   - **완전 방어(권장 단계): proof를 TLS 채널(RFC 5705 exporter)에 결착** → nonce 강함을 왕복·상태 0으로. (TLS가 중계 종단 + 스택의 exporter 노출 필요)
5. **거버넌스**: B 축 — `aud` 강제, `role` 방향성, `force_disconnect`, 동시접속 한도.
6. **인가 보강**: A 축 — MQTT 토픽 `authority` 검증(감청 차단).
7. **위협 경계 명시**: 감염 단말 실시간 키 오용은 범위 밖(상위 세션 탈취 탐지 층).
8. **선결 검증**: B의 Q1~Q5(레거시 MQTT 적재 가능성) + TLS/ssl 실사용 교차검증을 실현성 게이트로.

---

## 한 줄 결론

> 두 문서 차이 = **"검증 서버가 T5 선점 재전송을 TLS에 위임(A·무왕복)하느냐, 서버 nonce로 자체 방어(B·1왕복)하느냐"** 하나. TLS+인증서 검증이 보장되면 A, 현 trust-all이면 B가 우세하며, 완전 방어는 **TLS 채널 바인딩**으로 둘의 장점(무왕복 + T5 봉쇄)을 동시에 취할 수 있다. 거버넌스·전송축은 택일이 아니라 통합 대상이다.
